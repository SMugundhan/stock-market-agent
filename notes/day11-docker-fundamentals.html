Concept 01

What Problem Does Docker Solve?

Real World Scenario
Think of shipping cargo before and after standard shipping containers. Before standardization, every ship and port used different equipment, so cargo was manually repacked at every transfer — loading took days and things got lost or broke. After standard containers, the same box works on any ship, truck, or crane: load once, transfer anywhere, zero repacking.

Docker is the standard shipping container for software.

Before Docker

"It works on my machine!" — server crashes on deploy because the developer has Python 3.11, the server has 3.9; Redis versions differ; PATH and env vars differ; it works on Mac, breaks on Linux.

After Docker

Code + Python 3.11 + Redis 7.0 + all dependencies + config, packaged as one unit. Run that unit anywhere — laptop, server, cloud — identical behavior every time.

Concept 02

Core Docker Concepts

Real World Scenario
Use the analogy of a restaurant franchise. 
Docker Image     = The franchise recipe book + kitchen blueprint
                   Read-only. Never changes once built.

Docker Container = One actual running restaurant built from that blueprint
                   Many independent restaurants from the same blueprint.
                   Stopping a container doesn't delete the image.

Volume            = The restaurant's storage room
                   Data that must survive even if the restaurant is
                   rebuilt from the blueprint. Redis data, logs, uploads.

Network           = The internal phone system between restaurants
                   Containers can't talk to each other WITHOUT a network.

Image vs Container — the key difference

Image

Static snapshot, like a template. Built once from a Dockerfile. Stored on disk, shareable via Docker Hub. Running an image creates a container.

Container

A running instance of an image — a live process. Many containers can spawn from one image, each with its own filesystem and network.

Image = a recipe. Container = a dish made from that recipe. The recipe never changes; you can cook it many times.

Concept 03

What is a Dockerfile?

Real World Scenario
A Dockerfile is a very precise packing checklist for moving to a new city: start with an empty apartment (base image), install furniture (OS dependencies), unpack boxes (copy code), install appliances (pip install), set up your address (env config), note the door to use (expose port), and describe how to start living there (CMD). 
# ── Base image ──────────────────────────────────
FROM python:3.11-slim
# "slim" = minimal Linux with Python only, saves ~400MB

# ── Working directory ───────────────────────────
WORKDIR /app

# ── Copy dependency file first (cache-friendly) ─
COPY requirements.txt .

# ── Install dependencies ────────────────────────
RUN pip install --no-cache-dir -r requirements.txt

# ── Copy application code ───────────────────────
COPY . .

# ── Expose port ──────────────────────────────────
EXPOSE 8000

# ── Start command ────────────────────────────────
CMD ["uvicorn", "api.main:app", "--host", "0.0.0.0", "--port", "8000"]
Concept 04

Layer Caching — Why Order in a Dockerfile Matters

Real World Scenario
Think of building with LEGO bricks from a set of instructions. Each instruction is one layer, cached after the first build. Change step 7 and only steps 7+ rebuild; change step 2 and everything from step 2 onward rebuilds.

Code copied first — slow

COPY . .
RUN pip install ...

Code changes on every edit → pip install re-runs on every build.

Requirements copied first — fast

COPY requirements.txt .
RUN pip install ...
COPY . .

pip layer stays cached; only the final copy layer rebuilds most of the time.

Concept 05

Docker Compose

Real World Scenario
Your application isn't one thing — it's a FastAPI app and Redis. Docker Compose is the conductor for an orchestra: it starts every service together, wires up networking, and manages volumes from a single file. 
# docker-compose.yml
services:
  api:
    build: .
    ports:
      - "8000:8000"

  redis:
    image: redis:7
    ports:
      - "6379:6379"
docker compose up
# Starts EVERYTHING. Wires them together. Done.

Create .dockerignore

.env
.git

*.egg-info/
.pytest_cache/

Without it, Docker copies everything — including .git (hundreds of MB) and .env (a security risk). With it, only real application code gets copied → smaller, faster, more secure images.

03

Create Dockerfile

# ── Base Image ──────────────────────────
FROM python:3.11-slim

# ── System dependencies ─────────────────
RUN apt-get update && apt-get install -y \
    gcc \
    && rm -rf /var/lib/apt/lists/*

# ── Working directory ────────────────────
WORKDIR /app

# ── Install dependencies (cached layer) ──
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt

# ── Copy application code ────────────────
COPY . .

# ── Environment variables with defaults ──
ENV PYTHONUNBUFFERED=1
ENV PYTHONDONTWRITEBYTECODE=1

# ── Expose port ───────────────────────────
EXPOSE 8000

# ── Start command ─────────────────────────
CMD ["uvicorn", "api.main:app", \
     "--host", "0.0.0.0", \
     "--port", "8000", \
     "--workers", "1"]
04

Create docker-compose.yml

version: "3.9"

services:
  api:
    build:
      context: .
      dockerfile: Dockerfile
    container_name: stock-agent-api
    ports:
      - "8000:8000"
    environment:
      - GROQ_API_KEY=${GROQ_API_KEY}
      - TAVILY_API_KEY=${TAVILY_API_KEY}
      - REDIS_URL=redis://redis:6379
      - APP_VERSION=1.0.0
    depends_on:
      redis:
        condition: service_healthy
    volumes:
      - ./logs:/app/logs
    restart: unless-stopped
    networks:
      - stock-network

  redis:
    image: redis:7-alpine
    container_name: stock-agent-redis
    ports:
      - "6379:6379"
    volumes:
      - redis-data:/data
    healthcheck:
      test: ["CMD", "redis-cli", "ping"]
      interval: 10s
      timeout: 5s
      retries: 3
    restart: unless-stopped
    networks:
      - stock-network

volumes:
  redis-data:

networks:
  stock-network:
    driver: bridge

Note redis://redis:6379 — not localhost. Inside a Docker network, containers find each other by service name.

05

Update core/config.py for Docker

import os
from dotenv import load_dotenv

load_dotenv()

def get_version() -> str:
    try:
        with open("VERSION", "r") as f:
            return f.read().strip()
    except FileNotFoundError:
        return os.getenv("APP_VERSION", "0.1.0")

class Config:
    GROQ_API_KEY   = os.getenv("GROQ_API_KEY")
    TAVILY_API_KEY = os.getenv("TAVILY_API_KEY")

    REDIS_URL = os.getenv("REDIS_URL", "redis://localhost:6379")
    # local:  redis://localhost:6379
    # Docker: redis://redis:6379 (injected via compose)

    MODEL_NAME  = "llama-3.1-8b-instant"
    MAX_TOKENS  = 1024
    APP_VERSION = get_version()

config = Config()

docker exec -it container bash is the most useful debug command — it drops you into a shell inside a running container, exactly like SSH into a server.

Interview Prep

Interview Questions — Day 11

Q1 What problem does Docker solve and why does it matter for AI/ML projects specifically?
Docker solves the "works on my machine" problem by packaging code together with its exact runtime environment — Python version, system libraries, dependencies — into a portable container. This matters even more for ML projects because dependency chains (numpy, scipy, torch) interact tightly with each other and the OS. Docker ensures a model tested on a developer's Mac runs identically on a Linux cloud server with zero environment debugging.
Q2 What's the difference between a Docker Image and a Docker Container?
An image is a static, read-only blueprint — built once from a Dockerfile, stored on disk, shareable. A container is a live running instance created from an image, like a process. The same image can spawn multiple independent containers. Stopping a container leaves it intact but not running; removing it deletes it, but the image remains. It's the difference between a recipe and a dish cooked from that recipe.
Q3 Why does the order of commands in a Dockerfile matter?
Docker builds images in layers, each cached individually. If a layer changes, every layer after it rebuilds. Copying requirements.txt and running pip install before copying application code keeps the expensive install step cached and skipped on most builds, since code changes constantly but dependencies rarely do — making iterative development much faster.
Q4 Why is the Redis URL redis://redis:6379 instead of redis://localhost:6379?
Inside a Docker network, containers find each other by service name, not localhost. Localhost inside the api container refers to itself, not Redis. Docker Compose creates a virtual network where each service is reachable by its service name — so "redis" resolves to the Redis container's IP automatically. config.py reads REDIS_URL from an environment variable so the same code works locally and in Docker without changes.
Q5 What is a Docker Volume and why do you need one for Redis?
A volume is persistent storage that exists outside a container's lifecycle. Without one, data written inside a container is lost when the container is removed or recreated. Redis periodically saves its cache to /data — the named volume redis-data persists that data independently, so the cache survives restarts, deployments, and even image rebuilds.
Q6 What does --host 0.0.0.0 mean in the CMD, and why is it required in Docker?
By default uvicorn binds to 127.0.0.1, accepting connections only from the same machine — inside a container, that means only from inside the container itself. 0.0.0.0 means "listen on all network interfaces," so the container accepts traffic through Docker's virtual network interface. That's what makes the Compose port mapping work — host localhost:8000 can reach the container's port 8000.
