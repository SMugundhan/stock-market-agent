Core Ideas — Day 8

C01

What is a REST API?

▼
🌍 REAL WORLD — RESTAURANT AND ITS MENU You (customer) = Frontend / Client · Waiter = API · Kitchen = Your LangGraph agents

You don't walk into the kitchen and cook yourself. You tell the waiter what you want (a request). The waiter brings it to the kitchen, kitchen prepares it, waiter brings back your food (a response).

An API is the waiter — a defined way for two systems to talk without one needing to know the internal details of the other.

REST = Representational State Transfer. A set of conventions for designing APIs:

rest_conventions.txttext
1. Resources are nouns      → /stocks/AAPL  (not /getStockAAPL)
2. HTTP methods are verbs   → GET, POST, PUT, DELETE
3. Stateless                → Each request is independent
4. JSON for data exchange   → Standard format both sides understand

Bad API design (not RESTful):
  /getStockPrice?ticker=AAPL
  /createNewAnalysisForStock

Good API design (RESTful):
  GET    /stocks/AAPL     → get AAPL data
  POST   /analysis         → create new analysis
  DELETE /analysis/{id}    → delete an analysis

HTTP Methods and Status Codes

▼
🌍 REAL WORLD — REQUESTS YOU MAKE AT A BANK GET → "Show me my account balance" (just reading, no changes)
POST → "Open a new account" (creating something new)
PUT → "Update my address" (replacing existing data)
DELETE → "Close my account" (removing something)

Methods you'll actually use in your system:

GET  /stocks/{ticker}/quick     → get just price data
POST /analyze                   → run full multi-agent analysis
                                   (POST because it triggers a complex

▼

2. Checks their ID and paperwork is correct (Pydantic validation)
3. Directs them to the right department (your functions)
4. Hands back the response in proper format (JSON)
5. Keeps a logbook of who came when (auto-generated Swagger docs)

C04

Path Operations, Request & Response Models

▼
🌍 REAL WORLD — DOCTOR'S CLINIC FORMS Request Form (what YOU fill in): Name, Age, Symptoms — fields with defined types. You can't write "purple" in the Age field.
Response Form (what DOCTOR gives back): Diagnosis, Prescription, Follow-up — defined structure both sides agree on.

That's exactly what Pydantic models do for FastAPI.
endpoint_anatomy.pypython
from fastapi import FastAPI
from pydantic import BaseModel

app = FastAPI()

# ── Request Model (the "form" the user fills) ──────────
class AnalysisRequest(BaseModel):
    ticker:     str          # REQUIRED field
    query_type: str = "full"  # OPTIONAL — default "full"

# ── Response Model (the "form" we give back) ───────────
class AnalysisResponse(BaseModel):
    ticker:         str
    recommendation: str
    confidence:     float

# ── Path Operation (the actual endpoint) ───────────────
@app.post("/analyze", response_model=AnalysisResponse)
# @app.post = responds to POST requests at "/analyze"
# response_model = validates and formats OUR response

def analyze_stock(request: AnalysisRequest):
    # request is automatically validated BEFORE this function runs
    # FastAPI rejects invalid requests with 422 automatically
    ticker = request.ticker

    return AnalysisResponse(
        ticker=ticker,
        recommendation="BUY",
        confidence=0.78
    )
    # FastAPI converts this to JSON automatically

api/schemas.pypython
from pydantic import BaseModel, Field
from typing import Optional, List

# ── REQUEST MODELS ──────────────────────────────────────

class AnalysisRequest(BaseModel):
    """What the user sends us to request an analysis"""

    ticker: str = Field(
        ...,
        # ... means REQUIRED (no default value)
        description="Stock ticker symbol e.g. AAPL, TSLA",
        examples=["AAPL"]
        # examples shows up in Swagger UI as a sample
    )
    query_type: str = Field(default="full", description="full | quick | risk_only")
    session_id: Optional[str] = Field(default="anonymous", description="Session ID for conversation memory")


# ── RESPONSE MODELS (nested — mirrors Report Agent's JSON output) ──

class MarketData(BaseModel):
    current_price:    Optional[float] = None
    price_change_pct: Optional[float] = None
    rsi:              Optional[float] = None

class SentimentData(BaseModel):
    score:     Optional[float] = None
    headlines: List[str] = []

class RiskData(BaseModel):
    volatility_annual: Optional[float] = None
    risk_level:        Optional[str]   = None

class Verdict(BaseModel):
    recommendation: Optional[str]   = None
    confidence:     Optional[float] = None
    reasoning:      Optional[str]   = None
    key_factors:    List[str]       = []

class AnalysisResponse(BaseModel):
    """What we send back to the user"""
    ticker:            str
    report_date:       str
    market_data:       MarketData
    sentiment:         SentimentData
    risk:              RiskData
    verdict:           Verdict
    executive_summary: str
    disclaimer:        str

class HealthResponse(BaseModel):
    status:          str
    redis_connected: bool
    version:         str = "1.0.0"   # ← Day 8; see Future Improvements for dynamic version

class ErrorResponse(BaseModel):
    error:  str
    detail: Optional[str] = None

Why nested models? Notice AnalysisResponse contains MarketData, SentimentData etc. as sub-models — mirroring the exact JSON structure your Report Agent already produces, making the conversion seamless.

@app.get("/health", response_model=HealthResponse, tags=["Health"])
def health_check():
    """
    Health check endpoint.
    Used by monitoring, load balancers, Docker healthchecks
    to verify service is alive and dependencies are connected.
    """
    redis_ok = True
    try:
        redis_client.ping()
    except Exception:
        redis_ok = False

    return HealthResponse(
        status="healthy" if redis_ok else "degraded",
        redis_connected=redis_ok
    )


@app.post("/analyze", response_model=AnalysisResponse, tags=["Analysis"])
def analyze_stock(request: AnalysisRequest):
    """
    Main endpoint — runs full multi-agent stock analysis.
    Orchestrator → Price → News → Risk → Analyst → Report
    """
    print(f"\n🌐 API Request: {request.ticker} ({request.query_type})")

    initial_state = {
        "ticker": request.ticker, "query_type": request.query_type,
        "analysis_type": None, "should_fetch_news": None,
        "should_calculate_risk": None, "current_price": None,
        "price_change_pct": None, "rsi": None,
        "news_headlines": None, "sentiment_score": None,
        "volatility": None, "risk_level": None,
        "recommendation": None, "confidence": None,
        "reasoning": None, "final_report": None,
        "errors": [], "retry_count": 0
    }

    try:
        final_state = graph.invoke(initial_state)
    except Exception as e:
        raise HTTPException(status_code=500, detail=f"Analysis pipeline failed: {str(e)}")

    try:
        report = json.loads(final_state.get("final_report", "{}"))
    except json.JSONDecodeError:
        raise HTTPException(status_code=500, detail="Failed to parse analysis report")

    if final_state.get("analysis_type") == "error":
        raise HTTPException(
            status_code=404,
            detail=final_state.get("errors", ["Unknown error"])[0]
        )

    save_conversation_turn(request.session_id, request.ticker, report)
    return report


@app.get("/stocks/{ticker}/quick", tags=["Analysis"])
def quick_price_check(ticker: str):
    """
    Lightweight endpoint — just price data, skips news/risk/LLM calls.
    Much faster than /analyze for simple price checks.
    """
    initial_state = {
        "ticker": ticker.upper(), "query_type": "quick",
        # ... rest of None fields ...
        "errors": [], "retry_count": 0
    }
    final_state = graph.invoke(initial_state)
    return {
        "ticker":     ticker.upper(),
        "price":      final_state.get("current_price"),
        "change_pct": final_state.get("price_change_pct"),
        "rsi":        final_state.get("rsi")
    }

def get_version() -> str:
    """Read version from VERSION file, fallback to env var, fallback to default"""
    try:
        with open("VERSION", "r") as f:
            return f.read().strip()
    except FileNotFoundError:
        return os.getenv("APP_VERSION", "0.1.0")

class Config:

FUTURE

lifespan — Explicit startup/shutdown hooks instead of import-time globals

▼

Currently graph = build_graph() runs at import time. The proper pattern is FastAPI's lifespan context manager — gives you explicit startup/shutdown hooks so resources are initialized in a controlled, intentional way.

WHY --reload TRIGGERS THIS --reload is a dev-only flag. It restarts the whole process when you save a file, so build_graph() re-runs — this is intended dev behavior, not a bug. In production you run uvicorn api.main:app --workers 4 (no --reload) — each worker loads the graph exactly once at startup via lifespan.
api/main.py — with lifespan (Day 16+)python
from contextlib import asynccontextmanager
from fastapi import FastAPI

app_state = {}  # avoids polluting module-level globals

@asynccontextmanager
async def lifespan(app: FastAPI):
    """
    Code before 'yield' = STARTUP (runs once, controlled)
    Code after  'yield' = SHUTDOWN (runs on graceful exit)
    """
    # ── STARTUP ──────────────────────────────────────────
    print("🚀 Starting up: building graph...")
    app_state["graph"] = build_graph()

    print("🔌 Checking Redis connection...")
    try:
        redis_client.ping()
        print("✅ Redis connected")
    except Exception:
        print("⚠️  Redis unreachable — will degrade gracefully")

    yield
    # ── SHUTDOWN ─────────────────────────────────────────
    print("🛑 Shutting down: cleaning up...")
    redis_client.close()

app = FastAPI(title="...", version=config.APP_VERSION, lifespan=lifespan)

# In endpoints, read from app_state instead of bare global:
def analyze_stock(request: AnalysisRequest):
    graph = app_state["graph"]   # ← not a module-level global anymore

FUTURE

ASGI vs WSGI — What "ASGI server" actually means

▼
🌍 REAL WORLD — RESTAURANT SERVING STYLE WSGI (old): One waiter, one table at a time. Waiter takes Table 1's order and WAITS at the kitchen until food is ready. Table 2 starves the whole time.

ASGI (new): One waiter, juggles multiple tables. Takes Table 1's order, gives ticket to kitchen, immediately goes to Table 2. When Table 1's food is ready, serves it. No table blocks another.
TermSimple Meaning
WSGIOld standard — one request fully blocks the worker. Used by Flask, Django traditionally.
ASGINew standard — workers can juggle multiple requests while waiting on network calls.
uvicornThe actual ASGI server program that runs your FastAPI app.
FastAPIThe framework that defines routes/validation — needs an ASGI server like uvicorn to run.
lifespanExplicit startup/shutdown hooks — replaces scattered import-time side effects.
--reloadDev-only flag — restarts whole process on file save. Never used in production.
HONEST CATCH IN YOUR CURRENT CODE Your agents (price_agent_node, news_agent_node etc.) are synchronous functions with synchronous libraries (yfinance, requests-based clients). Even though uvicorn IS an ASGI server, FastAPI automatically offloads sync functions to a thread pool — you get reasonable concurrency without rewriting everything as async. True end-to-end async (using httpx.AsyncClient + async def nodes) is a Day 16+ refinement for maximum throughput under heavy load.
INTERVIEW

Q&A — Day 8

Q1What is a REST API and what makes an API "RESTful"?▼
REST stands for Representational State Transfer — a set of conventions for API design. A RESTful API treats data as resources accessed via nouns in the URL (/stocks/AAPL rather than /getStock), uses HTTP methods as verbs (GET for reading, POST for creating), is stateless meaning each request is independent, and exchanges data in JSON. This creates predictable, standardized interfaces that any client can consume.
Q2Why did you choose FastAPI over Flask or Django for this project?▼
FastAPI provides automatic request validation through Pydantic models, automatic interactive documentation via Swagger UI, and built-in async support — all critical for an AI agent API where requests can be slow due to LLM calls. Flask requires manually implementing validation and docs. Django is heavyweight with ORM and admin features we don't need for a pure API. FastAPI is purpose-built for exactly this use case and is the de facto standard for ML/AI APIs in production.
Q3Explain the role of Pydantic models in your FastAPI endpoints.▼
Pydantic models define the exact shape of data flowing in and out of each endpoint. AnalysisRequest validates incoming data — ensuring ticker is a string and is provided — before my function code even executes, automatically returning a 422 error for malformed requests. AnalysisResponse ensures what I return matches a defined contract, which also generates accurate API documentation automatically. This eliminates an entire category of bugs from inconsistent data shapes.
Q4Walk me through what happens when a POST request hits your /analyze endpoint.▼
FastAPI first validates the incoming JSON against AnalysisRequest — if ticker is missing, it immediately returns 422 without running my code. If valid, it creates an AnalysisRequest object and passes it to my function. I build the initial LangGraph state, invoke the graph which runs all 5 agents sequentially, parse the resulting JSON report, save it to conversation memory in Redis, and return it. FastAPI then validates my return value against AnalysisResponse and serializes it to JSON for the client.
Q5Why is there a separate /stocks/{ticker}/quick endpoint instead of just /analyze?▼
Different use cases have different performance needs. A simple price check doesn't need to wait for Tavily search calls and three separate LLM invocations — that would be wasteful latency and cost. By exposing query_type as a parameter and providing a dedicated quick endpoint, I let clients choose the right tool for their need — similar to how you wouldn't order a five-course meal when you just want a coffee.
Q6How would you handle a scenario where 1000 users hit /analyze simultaneously?▼
First, the Redis caching layer absorbs significant load since repeated tickers within the cache TTL window return instantly. Second, uvicorn can run multiple worker processes via --workers flag to handle concurrent requests across CPU cores. Third, for true scale I'd deploy behind a load balancer with multiple API instances, and make LLM calls within agents async to avoid blocking. Finally, rate limiting per user would prevent any single client from monopolizing the LLM API quota shared across all users.
