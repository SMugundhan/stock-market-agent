C02

What is Redis?

▼
🌍 REAL WORLD — WHITEBOARD VS FILING CABINET A filing cabinet (PostgreSQL) is organized and permanent, but slower. A whiteboard (Redis) is quick to write and read, temporary, and lives in RAM instead of on disk — about 1000× faster access.
🗄️

Filing Cabinet — PostgreSQL

  • ✓ Organized, permanent records
  • ✓ Good for complex relationships
  • ✓ Access time: milliseconds
⚡

Whiteboard — Redis

  • ✓ "What are we working on RIGHT NOW"
  • ✓ Data lives in RAM, not disk
  • ✓ Access time: microseconds

Redis = Remote Dictionary Server. Essentially a giant, super-fast key-value dictionary in memory.

C03

Short-Term vs Long-Term Memory

▼
🌍 REAL WORLD — HOW YOUR BRAIN HANDLES MEMORY Short-term (working memory): "What did they just say 2 sentences ago?" — disappears after the conversation ends, limited capacity.
Long-term: "What's my friend's birthday?" — persists for years, larger capacity but slower to search.
💬

Short-Term — Conversation Memory

  • Current conversation's context
  • "What ticker did we just discuss?"
  • Expires after ~1 hour of inactivity
  • Key: conversation:{session_id}
📦

Long-Term — Cache

  • Reusable data that doesn't change often
  • "AAPL's price 5 minutes ago" (no refetch needed)
  • Expires based on data freshness
  • Key: cache:price:{ticker}
ttl_examples.pypython
# Short-term: conversation context
redis.set(f"conversation:{session_id}", json.dumps(history), ex=3600)

# Long-term: cached analysis data

redis.set(f"cache:news:{ticker}", json.dumps(news_data), ex=900)
# ex=900 → expires in 900 seconds = 15 minutes (news changes slower)
C04

Caching and Why It Matters

▼
🌍 REAL WORLD — ORDERING FOOD AT A RESTAURANT YOU'VE VISITED BEFORE First visit: waiter explains the entire menu (slow, 5 minutes). Second visit same day: "I'll have what I had last time" (fast, 5 seconds) — you already know the menu.

Caching = storing the result of an expensive operation so you don't redo it.
WITHOUT CACHING — TWO USERS, SAME TICKER, 5 SECONDS APART
2:00:00
User A → yFinance + Tavily + LLaMA×3 → 8s, full API cost
2:00:05
User B → SAME calls AGAIN → another 8s, MORE API cost (wasteful!)
WITH CACHING — SAME SCENARIO
2:00:00
User A → fetches everything, SAVES to Redis (60s expiry) → 8s
2:00:05
User B → Redis HIT → 0.01s, $0 cost
CACHE HIT vs CACHE MISS
✅ Cache HIT — data found in Redis → return immediately (fast, free) HIT
❌ Cache MISS — data not in Redis → fetch fresh, then save to Redis MISS
cache_pattern.pypython
def get_price_with_cache(ticker: str):
    cache_key = f"cache:price:{ticker}"

    # Try cache first
    cached = redis_client.get(cache_key)
    if cached:
        print("✅ Cache HIT")
        return json.loads(cached)

    # Cache miss — fetch fresh data
    print("❌ Cache MISS — fetching fresh")
    price_data = fetch_from_yfinance(ticker)

    # Save to cache for next time
    redis_client.set(cache_key, json.dumps(price_data), ex=60)
    return price_data

def test_connection():
    """Quick test to verify Redis is reachable"""
    try:
        redis_client.ping()
        # .ping() sends a PING command, Redis responds PONG if alive
        print("✅ Redis connected successfully")
        return True
    except redis.ConnectionError:
        print("❌ Redis connection failed — is Redis running?")
        return False
memory/cache.pypython
import json
from memory.redis_client import redis_client

# Different data types need different expiration times
CACHE_TTL = {
    "price":       60,     # 60 seconds — price moves fast
    "news":        900,    # 15 minutes — news doesn't change that fast
    "risk":        1800,   # 30 minutes — risk metrics are stable
    "full_report": 120,    # 2 minutes — full analysis cache
}
# TTL = Time To Live — how long data stays valid in cache

def get_cached(key_type: str, ticker: str):
    """Returns parsed data if cache HIT, None if cache MISS"""
    cache_key = f"cache:{key_type}:{ticker}"
    # e.g. "cache:price:AAPL"

    cached_value = redis_client.get(cache_key)
    # .get() returns None if key doesn't exist or has expired

    if cached_value:
        print(f"✅ Cache HIT: {cache_key}")
        return json.loads(cached_value)
        # json.loads() converts stored JSON string back to Python dict

    print(f"❌ Cache MISS: {cache_key}")
    return None


def set_cached(key_type: str, ticker: str, data: dict):
    """Save data to cache with appropriate expiration"""
    cache_key = f"cache:{key_type}:{ticker}"
    ttl = CACHE_TTL.get(key_type, 300)
    # .get() with default 300 (5 min) if key_type not in our dict

    redis_client.set(
        cache_key,
        json.dumps(data),
        # json.dumps() converts Python dict → JSON string
        # Redis can only store strings, not Python objects directly
        ex=ttl
        # ex = expiration in seconds — auto-deleted after ttl seconds
    )
    print(f"💾 Cached: {cache_key} (expires in {ttl}s)")
agents/price_agent.py — updated with cachingpython
import yfinance as yf
from core.state import StockAnalysisState
from memory.cache import get_cached, set_cached
# ← NEW imports

def price_agent_node(state: StockAnalysisState) -> dict:
    ticker = state["ticker"]
    print(f"📈 Price Agent: Fetching data for {ticker}...")

    # ── STEP 1: Check cache first ────────────────────────
    cached_data = get_cached("price", ticker)

    if cached_data:
        # Cache HIT — return immediately, skip yFinance call entirely
        print(f"⚡ Using cached price data for {ticker}")
        return {
            "current_price":    cached_data["current_price"],
            "price_change_pct": cached_data["price_change_pct"],
            "rsi":              cached_data["rsi"],
            "errors": []
        }

    # ── STEP 2: Cache MISS — fetch fresh data ────────────
    try:
        stock = yf.Ticker(ticker)
        hist  = stock.history(period="1mo", interval="1d")

        current_price     = round(float(hist["Close"].iloc[-1]), 2)
        prev_price         = float(hist["Close"].iloc[-2])
        price_change_pct  = round(((current_price - prev_price) / prev_price) * 100, 2)

        closes   = hist["Close"]
        delta    = closes.diff()
        gain     = delta.clip(lower=0)
        loss     = -delta.clip(upper=0)
        avg_gain = gain.rolling(window=14).mean().iloc[-1]
        avg_loss = loss.rolling(window=14).mean().iloc[-1]

        rsi = 100.0 if avg_loss == 0 else round(100 - (100 / (1 + avg_gain/avg_loss)), 2)

        result = {"current_price": current_price, "price_change_pct": price_change_pct, "rsi": rsi}

        # ── STEP 3: Save to cache for next request ──────────
        set_cached("price", ticker, result)

        print(f"✅ Fresh data: ${current_price} | {price_change_pct}% | RSI: {rsi}")
        return {**result, "errors": []}

    except Exception as e:
        print(f"❌ Price Agent Error: {e}")
        return {"current_price": None, "price_change_pct": None,
                "rsi": None, "errors": [f"Price Agent failed: {str(e)}"]}
memory/conversation.pypython
import json
from datetime import datetime
from memory.redis_client import redis_client

CONVERSATION_TTL = 3600  # 1 hour of inactivity = conversation expires

def save_conversation_turn(session_id: str, ticker: str, report: dict):
    """Saves a single conversation turn so follow-up questions work"""
    conversation_key = f"conversation:{session_id}"

    # Get existing conversation history (or start fresh)
    existing = redis_client.get(conversation_key)
    history  = json.loads(existing) if existing else []

    history.append({
        "timestamp": datetime.now().isoformat(),
        "ticker":    ticker,
        "report_summary": {
            "recommendation": report.get("verdict", {}).get("recommendation"),
            "confidence":     report.get("verdict", {}).get("confidence"),
            "price":          report.get("market_data", {}).get("current_price"),
            "rsi":            report.get("market_data", {}).get("rsi"),
        }
    })

    # Keep only last 10 turns — prevent unlimited growth
    history = history[-10:]

    redis_client.set(conversation_key, json.dumps(history), ex=CONVERSATION_TTL)
    print(f"💬 Saved conversation turn for session {session_id}")


def get_conversation_history(session_id: str) -> list:
    """Retrieves conversation history for follow-up handling"""
    existing = redis_client.get(f"conversation:{session_id}")
    return json.loads(existing) if existing else []


def get_last_ticker(session_id: str) -> str | None:
    """For follow-ups like 'what was the RSI?' without re-specifying ticker"""
    history = get_conversation_history(session_id)
    if history:
        return history[-1]["ticker"]
        # [-1] = most recent entry
    return None

INTERVIEW

Q&A — Day 7

Q1Why does an agent system need memory if LangGraph already has State?▼
LangGraph's State only exists during a single graph.invoke() call — it's destroyed once the function returns. Memory via Redis persists across multiple separate invocations and even across server restarts. State is like working memory for one task; Redis memory is like a notebook that survives between tasks, enabling follow-up questions and avoiding redundant work.
Q2What is Redis and why is it preferred over a regular database for this use case?▼
Redis is an in-memory key-value store, meaning data lives in RAM rather than on disk, making reads and writes extremely fast — sub-millisecond versus milliseconds for traditional databases. It's preferred here because our use case is temporary, fast-changing data — cached prices, recent conversations — not permanent records requiring complex queries or relationships, which is where a database like PostgreSQL would be better.
Q3Explain the difference between short-term and long-term memory in your system.▼
Short-term memory is conversation context — what ticker we just discussed, recent turns — stored with a 1-hour expiration tied to session_id. Long-term memory in this context is actually our cache layer — reusable computed data like price or news that doesn't need to be refetched repeatedly, with expiration tuned to how fast that data type changes, like 60 seconds for price versus 15 minutes for news.
Q4What is a cache hit versus a cache miss, and how does TTL factor in?▼
A cache hit means requested data is found in Redis and returned immediately without recomputing. A cache miss means the data isn't there or has expired, so we fetch fresh data and then store it in cache for next time. TTL, or Time To Live, controls how long cached data remains valid — I use different TTLs per data type since price data goes stale in 60 seconds while risk metrics remain valid for 30 minutes.
Q5How would caching reduce costs in a production deployment of this system?▼
Every uncached request calls yFinance, Tavily, and makes 3 separate LLaMA API calls. If 100 users ask about AAPL within the same minute, without caching that's 100 times the API costs and rate limit consumption. With a 60-second price cache and 15-minute news cache, only the first request actually hits external APIs — the other 99 get instant cached responses, dramatically reducing both cost and latency.
Q6What happens if Redis goes down? How resilient is your system?▼
My cache functions use redis_client.get() which would raise a ConnectionError if Redis is unreachable. In production I would wrap cache calls in try/except blocks so a Redis outage degrades gracefully to cache misses — meaning slower responses since everything gets freshly fetched, but the system stays functional rather than crashing. Redis should be treated as a performance optimization, never a hard dependency for correctness.
