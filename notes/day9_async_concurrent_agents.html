Core Ideas — Day 9

C01

What Problem Does Async Actually Solve?

▼
🌍 REAL WORLD — ORDERING AT 3 FOOD STALLS AT A FESTIVAL Synchronous (what you have now): Walk to Stall A → order → WAIT 5 min standing there → get food. Walk to Stall B → WAIT 4 min. Walk to Stall C → WAIT 3 min. Total: 5+4+3 = 12 minutes.

Async: Walk to Stall A → place order → they say "5 min, we'll call you". WHILE WAITING → walk to Stall B → place order → "4 min". Walk to Stall C → "3 min". Collect each as ready. Total: max(5,4,3) = 5 minutes.

You're not cooking faster. You're just not standing around uselessly waiting.

While your code waits on a network response (yFinance, Tavily, Groq), it can let other waiting tasks proceed instead of blocking everything. That's the entire idea behind async/await.

C02

I/O-Bound vs CPU-Bound Work

▼
🌍 REAL WORLD — WAITING FOR WHATSAPP vs DOING MENTAL MATH I/O-bound = waiting on something external (WhatsApp reply) — your brain is basically idle.
CPU-bound = your own brain actually computing (mental arithmetic) — your brain is genuinely busy.

✅ I/O-Bound — async helps here

  • → Waiting for yFinance to respond
  • → Waiting for Tavily search results
  • → Waiting for Groq LLM to generate
  • → Your CPU is basically idle, just waiting

❌ CPU-Bound — async does NOT help

  • → np.std(daily_returns) — pure math
  • → drawdown_series.cummax() — pure math
  • → json.dumps(report_dict) — string processing
  • → CPU is actually busy doing work
IMPORTANT INTERVIEW NUANCE Async isn't a magic speed button for everything — it specifically helps when your code spends time waiting rather than computing. The numpy calculations in your Risk Agent run at the same speed whether you're in an async function or not.
C03

async/await Syntax, Explained Simply

▼
🌍 REAL WORLD — "I'M GOING TO WAIT IN THE QUEUE NOW — FEEL FREE TO HELP SOMEONE ELSE" await is exactly that: "pause me here, let other tasks run, resume me when this network call finishes."
sync_vs_async.pypython
# Regular (synchronous) function
def fetch_price(ticker):
    response = requests.get(url)   # ← BLOCKS here, nothing else can happen
    return response.json()

# Async function
async def fetch_price(ticker):
    response = await client.get(url)   # ← "pause me here, let other
                                          #    tasks run, resume when done"
    return response.json()

The three keywords you need:

keywords.pypython
async def my_function():
    # 'async def' = "this function CAN be paused and resumed"
    # It does NOT run in the background automatically

    result = await some_async_operation()
    # 'await' = "pause here, let event loop run other tasks meanwhile"

    return result

# To actually RUN an async function:
my_function()              # ❌ Just creates a coroutine object, doesn't run!
await my_function()       # ✅ Runs it (from inside another async function)
asyncio.run(my_function()) # ✅ Runs it from regular (sync) code

# Common beginner mistake:
result = get_data()         # ❌ Prints: <coroutine object...> NOT the value!
result = await get_data()  # ✅ Correct
C04

asyncio.gather() — Running Things Concurrently

▼
🌍 REAL WORLD — BACK TO THE 3 FOOD STALLS asyncio.gather() is literally you saying: "Place all 3 orders now, and tell me once ALL of them are ready."
gather_example.pypython
import asyncio

async def stall_a():
    await asyncio.sleep(5)
    return "Burger"

async def stall_b():
    await asyncio.sleep(4)
    return "Pizza"

async def stall_c():
    await asyncio.sleep(3)
    return "Tacos"

async def main():
    # Sequential — adds up: 5 + 4 + 3 = 12 seconds
    burger = await stall_a()
    pizza  = await stall_b()
    tacos  = await stall_c()

    # Concurrent — takes max(5, 4, 3) = 5 seconds
    burger, pizza, tacos = await asyncio.gather(
        stall_a(),
        stall_b(),
        stall_c()
    )
    # gather() starts all 3, waits for ALL to finish,
    # returns results in the same order they were passed in

Convert Agents to Async + Run in Parallel

THE REAL CONSTRAINT — SYNC LIBRARIES yfinance and the Tavily Python SDK are synchronous libraries. "Converting to async" means running these sync calls inside a thread pool controlled by async code, so they don't block the main event loop. The actual yfinance call still blocks a thread — but doesn't block the event loop from starting other tasks.
core/async_utils.pypython
import asyncio
from functools import partial

async def run_sync_in_thread(func, *args, **kwargs):
    """
    Runs a synchronous (blocking) function inside a thread pool,
    so it doesn't block the asyncio event loop.

    This is the bridge between sync libraries (yfinance, tavily)
    and our async agent functions.
    """
    loop = asyncio.get_event_loop()
    # get_event_loop() gets the currently running event loop

    func_with_args = partial(func, *args, **kwargs)
    # partial() pre-fills the function's arguments so we can pass
    # it to run_in_executor without arguments

    return await loop.run_in_executor(None, func_with_args)
    # run_in_executor(None, ...) = "run this in the default thread pool"
    # 'await' here pauses THIS coroutine, but other coroutines
    # can keep running while this thread pool work happens
agents/price_agent.py — async versionpython
import yfinance as yf
from core.state import StockAnalysisState
from core.async_utils import run_sync_in_thread
from memory.cache import get_cached, set_cached

def _fetch_price_sync(ticker: str) -> dict:
    """
    The actual blocking yfinance logic — UNCHANGED from before.
    Stays a regular function — yfinance itself can't be async.
    """
    stock = yf.Ticker(ticker)
    hist  = stock.history(period="1mo", interval="1d")

    if hist.empty:
        raise ValueError(f"No price data for {ticker}")

    current_price    = round(float(hist["Close"].iloc[-1]), 2)
    prev_price       = float(hist["Close"].iloc[-2])
    price_change_pct = round(((current_price - prev_price) / prev_price) * 100, 2)

    closes   = hist["Close"]
    delta    = closes.diff()
    gain     = delta.clip(lower=0)
    loss     = -delta.clip(upper=0)
    avg_gain = gain.rolling(window=14).mean().iloc[-1]
    avg_loss = loss.rolling(window=14).mean().iloc[-1]
    rsi = 100.0 if avg_loss == 0 else round(100 - (100 / (1 + avg_gain/avg_loss)), 2)

    return {"current_price": current_price, "price_change_pct": price_change_pct, "rsi": rsi}


async def price_agent_node(state: StockAnalysisState) -> dict:
    # ↑ NOTICE: 'async def' instead of just 'def'
    ticker = state["ticker"]
    print(f"📈 Price Agent: Fetching {ticker}...")

    cached_data = get_cached("price", ticker)
    if cached_data:
        print(f"⚡ Cache HIT for {ticker}")
        return {**cached_data, "errors": []}

    try:
        # await the thread-pool-wrapped sync call
        # While yfinance waits in its thread, the event loop
        # is FREE to run News Agent or Risk Agent
        result = await run_sync_in_thread(_fetch_price_sync, ticker)
        set_cached("price", ticker, result)
        return {**result, "errors": []}

    except Exception as e:
        return {"current_price": None, "price_change_pct": None,
                "rsi": None, "errors": [f"Price Agent failed: {str(e)}"]}

    ticker = state["ticker"]
    print(f"📰 News Agent: Fetching news for {ticker}...")

    try:
        # Tavily call → runs in thread pool, doesn't block event loop
        articles = await run_sync_in_thread(_fetch_news_sync, ticker)

        if not articles:
            return {"news_headlines": [], "sentiment_score": 0.0, "errors": []}

        headlines, contents = [], []
        for article in articles:
            title   = article.get("title", "")
            content = article.get("content", "")[:200]
            if title:
                headlines.append(title)
                contents.append(f"{title}: {content}")

        sentiment_prompt = f"Analyze {ticker} sentiment:\n{chr(10).join(contents)}\nSENTIMENT_SCORE: [decimal -1.0 to 1.0]"

        # llm.invoke() is also sync — OR use llm.ainvoke() directly (true async!)
        response = await run_sync_in_thread(llm.invoke, sentiment_prompt)
        # Better option: response = await llm.ainvoke(sentiment_prompt)
        # langchain-groq supports true async — no thread pool needed for LLM calls

        sentiment_score = 0.0
        for line in response.content.strip().split("\n"):
            if line.startswith("SENTIMENT_SCORE:"):
                sentiment_score = float(line.split(":")[1].strip())

        return {"news_headlines": headlines, "sentiment_score": sentiment_score, "errors": []}

    except Exception as e:
        return {"news_headlines": [], "sentiment_score": 0.0,

from langgraph.graph import StateGraph, END
from core.state import StockAnalysisState
from agents.orchestrator  import orchestrator_node
from agents.price_agent   import price_agent_node
from agents.news_agent    import news_agent_node
from agents.risk_agent    import risk_agent_node
from agents.analyst_agent import analyst_agent_node
from agents.report_agent  import report_agent_node

def build_parallel_graph():
    graph = StateGraph(StockAnalysisState)

    graph.add_node("orchestrator",  orchestrator_node)
    graph.add_node("price_agent",   price_agent_node)
    graph.add_node("news_agent",    news_agent_node)
    graph.add_node("risk_agent",    risk_agent_node)
    graph.add_node("analyst_agent", analyst_agent_node)
    graph.add_node("report_agent",  report_agent_node)

    graph.set_entry_point("orchestrator")

    # ── FAN-OUT: orchestrator triggers all 3 simultaneously ──
    graph.add_edge("orchestrator", "price_agent")   # ← same source
    graph.add_edge("orchestrator", "news_agent")    # ← same source
    graph.add_edge("orchestrator", "risk_agent")    # ← same source
    # Because these are now 'async def', LangGraph runs them
    # CONCURRENTLY using its internal asyncio.gather() mechanism

    # ── FAN-IN: analyst waits for ALL 3 to complete ──────────
    graph.add_edge("price_agent", "analyst_agent")  # ← same dest
    graph.add_edge("news_agent",  "analyst_agent")  # ← same dest
    graph.add_edge("risk_agent",  "analyst_agent")  # ← same dest
    # LangGraph waits for all 3, merges their State updates,
    # THEN runs analyst_agent with the fully populated State

    graph.add_edge("analyst_agent", "report_agent")
    graph.add_edge("report_agent",  END)

    return graph.compile()

DEEP DIVES

Thread Pool vs True Async — OS Level

DEEP DIVE

What exactly changes between thread-pool async and true async?

▼

The physical network latency doesn't change — Yahoo Finance still takes the same 200ms to respond. What changes is what resource holds the wait and at what cost.

WHAT HOLDS THE WAIT — SIDE BY SIDE

⚡ Sync Wrapped in Thread Pool

Main Loop
free — doing other work
Thread A
BLOCKED waiting… (Price)
Thread B
BLOCKED waiting… (News)
Thread C
BLOCKED waiting… (Risk)
3 OS threads: ~4MB × 3 = ~12MB RAM wasted

✅ Truly Async (httpx + OS epoll)

Main Loop
free — doing other work
OS fd=5
kernel watching socket (Price)
OS fd=6
kernel watching socket (News)
OS fd=7
kernel watching socket (Risk)
0 extra threads: 3 fd registrations = ~300 bytes
OS LEVEL

Step-by-step trace — what the OS actually does for each approach

▼

Sync wrapped in thread pool — step by step:

thread_pool_trace.txttext
Step 1: Main event loop spawns OS Thread A, hands it yfinance call
Step 2: Thread A opens TCP socket, sends HTTP request to Yahoo Finance
Step 3: Yahoo Finance hasn't responded yet.
        Thread A does this:
          while True:
              check socket → no data yet (frozen, doing nothing)
              check socket → no data yet
              ...
        Thread A is 100% tied up, consuming ~4MB RAM, doing NOTHING
Step 4: Yahoo Finance responds. Thread A wakes, reads data, returns.
Step 5: Main event loop collects result from Thread A.

Truly async (httpx + epoll) — step by step:

true_async_trace.txttext
Step 1: Main event loop (ONLY THREAD!) calls httpx.
        httpx opens TCP socket, sends HTTP request — same as before.
Step 2: Yahoo Finance hasn't responded yet.
        httpx tells the OS: "Register fd=5. Wake me when data arrives."
        This is epoll (Linux) / kqueue (Mac) / IOCP (Windows).
        NO thread is spawned. NO thread is blocked.
        The OS kernel watches the socket at the hardware level.
Step 3: Main event loop is FREE.
        Same single thread now sends News request (fd=6 registered)
        and Risk request (fd=7 registered).
        Three requests in-flight on ONE thread.
Step 4: OS fires: "fd=7 ready" (Risk arrived first)
        Event loop resumes Risk Agent coroutine, processes result.
Step 5: OS fires: "fd=5 ready" (Price arrived)
        Event loop resumes Price Agent coroutine.
Step 6: OS fires: "fd=6 ready" (News arrived last)
        Event loop resumes News Agent coroutine.
SCALING

Why this matters — 3 users vs 500 users

▼
ScenarioThread PoolTrue Async
1 waiting task1 OS thread (~4MB RAM)1 file descriptor (~100 bytes)
3 concurrent agents3 OS threads (~12MB)3 fd registrations (~300 bytes)
500 users × 3 agents1500 OS threads (~6GB RAM)1500 fd registrations (~150KB)
Scale limit~10k threads (OS limit)Hundreds of thousands of fd's
🌍 NUMBERS TICKET ANALOGY Thread pool: 3 separate people each go stand in 3 separate queues. Works for 3 queues. Hire 1500 people for 1500 queues — gets expensive fast.

True async: ONE person takes a numbered ticket at all 3 counters, then sits down and reads a book. When counter calls their number, they get up, get served, sit back down. One person can hold tickets at 1000 counters — "holding a ticket" costs almost nothing.
💡 THE ONE-SENTENCE INTERVIEW ANSWER "Both free the main event loop during the wait — but sync-wrapped-as-async offloads the wait to an OS thread that sits blocked and consumes resources, while truly async registers the wait with the OS kernel via a file descriptor, which costs almost nothing and scales to thousands of concurrent operations on a single thread."

Where you actually have a true-async option right now

▼

INTERVIEW

Q&A — Day 9

Q1What problem does async/await actually solve, and what problem does it NOT solve?▼
Async solves the problem of wasted waiting time during I/O operations — while waiting for a network response from an external API, the program can work on other tasks instead of blocking. It does NOT make CPU-bound computations faster — calculating RSI or running numpy operations takes the same time whether you're in an async function or not, because the CPU still has to do the same amount of actual work.
Q2Why can't you just call an async function directly like a normal function?▼
Calling an async function returns a coroutine object, not the result — it's like creating a recipe but not cooking it. You need 'await' to actually execute it and get the result, and await only works inside another async function. To run an async function from regular synchronous code, you need asyncio.run(), which starts the event loop that actually drives execution.
Q3yfinance and Tavily's Python SDK are synchronous libraries. How did you make your agents async despite this?▼
I used asyncio's run_in_executor to run the blocking synchronous calls inside a thread pool, controlled by an async wrapper function. This means the actual yfinance call still blocks that one thread, but it doesn't block the main asyncio event loop — so other coroutines, like News Agent or Risk Agent's requests, can proceed concurrently while Price Agent's thread is waiting on Yahoo Finance's response.
Q4Explain asyncio.gather() and how it connects to LangGraph's fan-out pattern from Day 5.▼
asyncio.gather() takes multiple coroutines, starts them all concurrently, and waits until every single one completes before returning their combined results. This is the actual underlying mechanism behind LangGraph's fan-out/fan-in pattern — when a node has multiple outgoing edges to async node functions, LangGraph's execution engine runs them concurrently using a gather-like mechanism rather than waiting for each one sequentially.
Q5In your benchmark, why wasn't the parallel version exactly 3× faster even though you parallelized 3 agents?▼
Three reasons: first, Analyst and Report agents still run sequentially after the fan-in completes, since they depend on all prior results. Second, there's overhead in spinning up async tasks and thread pool execution. Third, the three parallel agents don't take exactly equal time — overall speedup is bounded by the slowest one (News Agent at ~5s), not divided evenly by three, since the others finish early and just wait.
Q6When would you NOT want to convert a function to async?▼
When the function is purely CPU-bound with no I/O waiting — like the RSI or volatility calculations using numpy — converting it to async adds complexity without any performance benefit, since there's no waiting period to exploit. Async is specifically valuable for I/O-bound code; forcing it onto CPU-bound code is over-engineering that makes the code harder to read for zero gain.
