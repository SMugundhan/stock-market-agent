C01

What is an Orchestrator Agent?

▼
🌍 REAL WORLD — AIR TRAFFIC CONTROLLER AT CHENNAI AIRPORT The ATC doesn't fly any plane themselves. Their job: decide which runway each plane uses, sequence the order of landings, reroute planes if weather changes, escalate to emergency protocol if needed, coordinate ground crew and gates.

The ATC doesn't do the work — they coordinate who does what and when. That's your Orchestrator Agent.

Without Orchestrator: Agents run blindly in a fixed sequence. Every request is treated the same regardless of what's actually needed.

With Orchestrator: The system adapts dynamically.

C02

What is Conditional Routing?

▼
🌍 REAL WORLD — HOSPITAL TRIAGE SYSTEM A patient comes to emergency. Triage nurse assesses them:
If vitals critical → ICU immediately · If injury severe → surgery
If condition moderate → general ward · If minor → outpatient

The nurse doesn't send everyone to the same place. They assess and route accordingly.

In LangGraph, a routing function looks at State and returns a string — the name of the next node. LangGraph routes accordingly.

ROUTING FUNCTION — AFTER ANALYST AGENT
route_after_analyst(state) → returns string
if errors exist
→ "error_handler"
if confidence < 0.4
→ "retry_analyst"
default (good result)
→ "report_agent"

# Regular edge — always goes to same place
graph.add_edge("price_agent", "news_agent")

# Conditional edge — routing function decides where to go
graph.add_conditional_edges(

    route_after_analyst,       # CALL this function
    {                          # MAP return values to node names
        "error_handler":  "error_handler",
        "retry_analyst":  "analyst_agent",  # loop back!
        "report_agent":   END
    }

C03

Sequential vs Parallel Execution

▼
🌍 REAL WORLD — RESTAURANT KITCHEN Sequential: Cook 1 finishes starter → Cook 2 starts main → Cook 3 starts dessert. Total: 30+40+20 = 90 mins.
Parallel: All 3 cooks work simultaneously. Total: max(30,40,20) = 40 mins.
EXECUTION COMPARISON — YOUR PIPELINE
⟶ SEQUENTIAL (current)

📰 News Agent5s

🧠 Analyst Agent2s
Total: 3+5+4+2 = 14s
⇉ PARALLEL (fan-out/fan-in)
FAN-OUT — runs simultaneously
📈 Price Agent3s
📰 News Agent5s
⚠️ Risk Agent4s
↓ fan-in: wait for all 3
🧠 Analyst Agent2s
Total: max(3,5,4)+2 = 7s ← 2× faster
WHY WE BUILD SEQUENTIAL TODAY Parallel adds complexity: state merging conflicts, harder error handling if one of 3 parallel agents fails, interleaved debug logs. Sequential is easier to explain in interviews and debug. We add parallel execution in a later day once all agents are tested individually.

Fan-Out / Fan-In Pattern — Deep Dive

▼
🌍 REAL WORLD — CRICKET TEAM MORNING PREP Coach gives briefing (fan-out — sends everyone their tasks).
Batsmen go to nets + Bowlers to bowling crease + Fielders do drills — all simultaneously.
Everyone assembles back for team meeting (fan-in — collect all results).
graph_parallel_conceptual.pypython
def build_parallel_graph():
    graph = StateGraph(StockAnalysisState)
    # add nodes ...

    # ── FAN-OUT ─────────────────────────────────────────────
    # Same source node → 3 outgoing edges = run ALL 3 in parallel
    graph.add_edge("orchestrator", "price_agent")  # ← same source
    graph.add_edge("orchestrator", "news_agent")   # ← same source
    graph.add_edge("orchestrator", "risk_agent")   # ← same source
    # LangGraph: "3 outgoing from orchestrator → spawn all 3"

    # ── FAN-IN ──────────────────────────────────────────────
    # Same destination → LangGraph WAITS for all 3 to finish
    graph.add_edge("price_agent", "analyst_agent")  # ← same dest
    graph.add_edge("news_agent",  "analyst_agent")  # ← same dest
    graph.add_edge("risk_agent",  "analyst_agent")  # ← same dest
    # LangGraph: "analyst_agent has 3 incoming → wait for ALL 3"
    # Then merges all 3 State updates before running analyst_agent

    graph.add_edge("analyst_agent", END)
    return graph.compile()
WHAT LANGGRAPH DOES INTERNALLY When LangGraph sees multiple edges FROM same node → spawns all targets as concurrent asyncio tasks. Each node writes to its own section of State. State is safely merged after ALL complete. When it sees multiple edges TO same node → waits for all incoming, merges their updates, then runs the target with the fully merged State.

from langchain_groq import ChatGroq
from core.state import StockAnalysisState
from core.config import config

llm = ChatGroq(api_key=config.GROQ_API_KEY, model_name=config.MODEL_NAME)

def orchestrator_node(state: StockAnalysisState) -> dict:
    """
    Entry point of the system.
    Reads:  state["ticker"], state["query_type"]
    Writes: analysis_type, should_fetch_news, should_calculate_risk
    """
    ticker = state.get("ticker", "")
    print(f"\n🎯 Orchestrator: Starting analysis for {ticker}")

    # ── Validate ticker ───────────────────────────────────
    if not ticker or len(ticker) > 10:
        # Tickers are always short — AAPL, TSLA, GOOGL etc.
        print("❌ Invalid ticker")
        return {
            "errors": [f"Invalid ticker: {ticker}"],
            "analysis_type": "error"
        }

    # ── Determine analysis type ───────────────────────────
    query = state.get("query_type", "full").lower()

    if query == "quick":
        # Quick — only price, skip news and risk
        print("⚡ Quick analysis mode")
        return {
            "analysis_type":         "quick",
            "should_fetch_news":     False,
            "should_calculate_risk": False,
            "errors": []
        }

    elif query == "risk_only":
        print("⚠️  Risk-only mode")
        return {
            "analysis_type":         "risk_only",
            "should_fetch_news":     False,
            "should_calculate_risk": True,
            "errors": []
        }

    else:
        # Full analysis — run everything
        print("🔍 Full analysis mode — running all agents")
        return {
            "analysis_type":         "full",
            "should_fetch_news":     True,
            "should_calculate_risk": True,
            "errors": []
        }

agents/routing.pypython
from core.state import StockAnalysisState

def route_after_orchestrator(state: StockAnalysisState) -> str:
    """
    Called after Orchestrator.
    Returns the name of the next node.
    """
    errors        = state.get("errors", [])
    analysis_type = state.get("analysis_type", "full")

    if errors:
        # Something went wrong — go to error handler
        print("🔀 Routing → error_handler (errors detected)")
        return "error_handler"

    # Default — run full pipeline starting with price agent
    print("🔀 Routing → price_agent (full mode)")
    return "price_agent"


def route_after_analyst(state: StockAnalysisState) -> str:
    """
    Called after Analyst Agent.
    Decides: confident enough? Or retry?
    """
    confidence = state.get("confidence", 0.0)
    errors     = state.get("errors", [])
    retries    = state.get("retry_count", 0)
    # retry_count tracks how many times we've already retried

    if errors and retries < 2:
        # Errors but haven't retried twice yet — retry once
        print(f"🔀 Routing → retry (attempt {retries + 1})")
        return "retry_analyst"

    if confidence < 0.4 and retries < 1:
        # Very low confidence on first try — retry once
        print(f"🔀 Routing → retry (low confidence: {confidence})")
        return "retry_analyst"

    # Good enough — go to report
    print(f"🔀 Routing → report_agent (confidence: {confidence})")
    return "report_agent"

from core.state import StockAnalysisState

def error_handler_node(state: StockAnalysisState) -> dict:
    """
    Called when something goes wrong in the pipeline.
    Returns a graceful error response instead of crashing.
    """
    errors = state.get("errors", [])
    ticker = state.get("ticker", "UNKNOWN")

    print(f"🚨 Error Handler: {errors}")

    # Create user-friendly error report
    error_report = f"""
Analysis for {ticker} encountered issues:

Errors: {', '.join(errors)}

Partial data available:
- Price: {state.get('current_price', 'unavailable')}
- Recommendation: Unable to complete due to errors

Please retry or check the ticker symbol.
    """.strip()

    return {
        "final_report":    error_report,
        "recommendation":  "UNAVAILABLE",
        "confidence":      0.0
    }
agents/graph.py — Day 5 full versionpython
from langgraph.graph import StateGraph, END
from core.state import StockAnalysisState

from agents.orchestrator  import orchestrator_node
from agents.price_agent   import price_agent_node
from agents.news_agent    import news_agent_node
from agents.risk_agent    import risk_agent_node
from agents.analyst_agent import analyst_agent_node
from agents.error_handler import error_handler_node
from agents.routing       import route_after_orchestrator, route_after_analyst

def build_graph():
    graph = StateGraph(StockAnalysisState)

    # ── Add all nodes ──────────────────────────────────────
    graph.add_node("orchestrator",  orchestrator_node)
    graph.add_node("price_agent",   price_agent_node)
    graph.add_node("news_agent",    news_agent_node)
    graph.add_node("risk_agent",    risk_agent_node)
    graph.add_node("analyst_agent", analyst_agent_node)
    graph.add_node("error_handler", error_handler_node)

    # ── Entry point — Orchestrator runs FIRST ──────────────
    graph.set_entry_point("orchestrator")

    # ── Conditional edge after Orchestrator ───────────────
    graph.add_conditional_edges(
        "orchestrator",
        route_after_orchestrator,
        {
            "price_agent":  "price_agent",   # normal route
            "error_handler": "error_handler"  # error route
        }
    )

    # ── Sequential edges for main pipeline ────────────────
    graph.add_edge("price_agent",   "news_agent")
    graph.add_edge("news_agent",    "risk_agent")
    graph.add_edge("risk_agent",    "analyst_agent")

    # ── Conditional edge after Analyst ────────────────────
    graph.add_conditional_edges(
        "analyst_agent",
        route_after_analyst,
        {
            "report_agent":  END,                # success → end
            "retry_analyst": "analyst_agent"     # retry → loop back
        }
    )
    # Note: END is used directly since Report Agent comes Day 7

    # ── Error handler always ends ──────────────────────────
    graph.add_edge("error_handler", END)

    return graph.compile()

Doubt 2 — The Simple Ticker Validation is Too Naive

Our len(ticker) > 10 check misses typos like "APL", full names like "TESLA", and Indian stock symbols. Here's the proper 3-layer solution.

D02

3-Layer Ticker Validation — Production Grade

▼
1

Basic Cleanup

Strip whitespace, uppercase, reject empty strings. Catches " AAPL " → "AAPL" and "" → error.

2

Company Name Mapping

Dictionary of common mistakes: "TESLA" → "TSLA", "apple" → "AAPL", "google" → "GOOGL". Handles 95% of user mistakes instantly, no API call needed.

3

Real-Time yFinance Validation

Actually hits Yahoo Finance API. If data comes back empty → invalid ticker. Uses Python's built-in difflib.get_close_matches() to suggest the closest valid ticker for typos like "APL" → suggests "AAPL".

orchestrator.py — 3-layer validationpython
import yfinance as yf
from difflib import get_close_matches
# difflib = Python built-in fuzzy matching library

COMMON_NAME_MAP = {
    "tesla":     "TSLA",
    "apple":     "AAPL",
    "microsoft": "MSFT",
    "google":    "GOOGL",
    "alphabet":  "GOOGL",
    "amazon":    "AMZN",
    "meta":      "META",
    "nvidia":    "NVDA",
    "reliance":  "RELIANCE.NS",  # Indian stocks need .NS suffix
    "infosys":   "INFY",
}

KNOWN_TICKERS = [
    "AAPL", "TSLA", "GOOGL", "AMZN", "MSFT",
    "META", "NFLX", "NVDA", "AMD",  "INTC",
    "JPM",  "BAC",  "GS",   "WMT",  "RELIANCE.NS", "INFY",
]

def validate_and_resolve_ticker(ticker: str) -> tuple[str, str | None]:
    """Returns (resolved_ticker, error_message)"""

    # ── Layer 1: Basic cleanup ────────────────────────────
    ticker = ticker.strip().upper()
    if not ticker:
        return "", "Ticker cannot be empty."

    # ── Layer 2: Company name mapping ─────────────────────
    if ticker.lower() in COMMON_NAME_MAP:
        resolved = COMMON_NAME_MAP[ticker.lower()]
        print(f"📝 Resolved '{ticker}' → '{resolved}'")
        return resolved, None

    # ── Layer 3: Real-time yFinance validation ─────────────
    try:
        test = yf.Ticker(ticker)
        info = test.info

        has_price = info.get("regularMarketPrice") is not None
        has_name  = info.get("longName") is not None

        if not has_price and not has_name:
            # Invalid — try fuzzy suggestion
            suggestions = get_close_matches(
                ticker, KNOWN_TICKERS,
                n=3, cutoff=0.5
            )
            # get_close_matches("APL", [...]) → ["AAPL"]
            if suggestions:
                return "", f"'{ticker}' not found. Did you mean: {', '.join(suggestions)}?"
            else:
                return "", f"Ticker '{ticker}' not found. Please check the symbol."

        print(f"✅ Validated: {ticker} = {info.get('longName', ticker)}")
        return ticker, None

    except Exception as e:
        return "", f"Could not validate '{ticker}': {str(e)}"

INTERVIEW

Q&A — Day 5

Q1What is an Orchestrator Agent and why do you need one?▼
An Orchestrator Agent coordinates other agents — it decides which agents to call, in what order, and what to do based on results. Without it, agents run blindly in a fixed sequence. With an Orchestrator, the system can adapt — run different agents for different request types, handle failures gracefully, retry low-confidence results. It's the difference between a rigid script and an intelligent system.
Q2What is Conditional Routing in LangGraph and how does it work?▼
Conditional routing uses a Python function that receives the current State and returns a string — the name of the next node. LangGraph uses add_conditional_edges() with a mapping dictionary of possible return values to node names. This enables dynamic flow — agents can loop back for retries, branch to error handlers, or skip unnecessary steps based on what they find in State.
Q3What's the difference between sequential and parallel agent execution?▼
Sequential execution runs agents one by one — each waits for the previous to finish. In my current pipeline, Price → News → Risk runs sequentially taking ~14 seconds total. Parallel execution uses fan-out/fan-in — the Orchestrator spawns multiple agents simultaneously, waits for all to complete (fan-in), then feeds combined results to the next stage. The same three agents would take only ~7 seconds total since they run concurrently.
Q4When would you use a loop in a LangGraph graph?▼
Loops are used for retry logic and self-correction. In my system, after the Analyst Agent runs, if confidence is below 0.4 (very uncertain), the routing function returns "retry_analyst" which loops back to the Analyst Agent. This gives the system a chance to self-correct before returning a low-quality result to the user. Without loops, you'd have to build separate retry infrastructure outside the graph.
Q5How does your system handle agent failures gracefully?▼
Each agent has a try/except block that catches errors and writes them to the errors list in State without crashing. The Orchestrator's routing function checks the errors field and routes to a dedicated Error Handler node instead of continuing the pipeline. The Error Handler generates a user-friendly message with whatever partial data was collected, rather than returning a raw Python exception to the user.
Q6How would you scale this system to handle multiple requests simultaneously?▼
Each graph invocation is stateless and independent — the State dict is created fresh per request. For concurrent requests, I would deploy the FastAPI layer with async endpoints using Python's asyncio, with multiple workers via uvicorn. LangGraph graphs are thread-safe per invocation. For production scale I would add a Redis-based job queue where requests are queued and processed by a pool of worker processes — each handling one graph invocation at a time.
