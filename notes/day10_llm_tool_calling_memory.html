Core Ideas — Day 10

C01

Rule-Based vs LLM-Driven Orchestration

▼
🌍 REAL WORLD — TWO TYPES OF CUSTOMER SERVICE MANAGERS Manager A (Rule-Based): Has a flowchart on their desk. Customer says "quick price check" → follows arrow to Box 2. Customer says something unexpected → manager is lost, no arrow for this.

Manager B (LLM-Driven): No flowchart needed. Actually READS and UNDERSTANDS the request. "Is AAPL too risky for a conservative investor?" → thinks: "This needs risk data and sentiment but not full technical analysis." Adapts dynamically to what's actually being asked.

The key difference: rule-based systems are brittle — they only handle cases you explicitly programmed. LLM-driven orchestration generalizes to cases you never anticipated.

📋 Rule-Based (Day 5)

  • if query_type == "quick" → price_agent
  • if query_type == "full" → all agents
  • YOU decided the logic, hardcoded in Python
  • Breaks on unexpected queries
  • Fast, predictable, no LLM cost

🤖 LLM-Driven (Day 10)

  • LLM reads the user's actual query
  • LLM decides which agents/tools to call
  • The LLM decides the logic at runtime
  • Handles any query you never programmed
  • Slower, costs tokens, but truly intelligent
C02

How LLM Tool Calling Actually Works

▼
🌍 REAL WORLD — NEW INTERN WITH A COMPANY DIRECTORY Manager (LLM) receives: "Find out if Apple's earnings are better than Samsung's and prepare a risk comparison." Manager sees available tools: get_stock_price, get_news, calculate_risk, get_earnings_report. Manager decides: I need calculate_risk and get_earnings_report for both, but NOT get_stock_price (question isn't about today's price). Calls tools in the right order with the right arguments.

When you define a tool for an LLM, you give it three things:

THE 3 PARTS EVERY TOOL NEEDS
1
NAME
What the LLM calls it by. Should be a clear, descriptive verb-noun: get_stock_price, calculate_risk
2
DESCRIPTION
Most critical. How the LLM knows WHEN to use it. Must say both when TO use it AND when NOT to.
3
PARAMETERS
What arguments the LLM must provide. Type hints are essential — the LLM uses them to know what format to send.
tool_flow.txttext
LLM receives:
  - User query: "Is AAPL too volatile for retirement portfolios?"
  - List of available tools with names + descriptions
  - Previous conversation context

LLM response is ONE of two things:
  Option A — Tool call: "I want to call calculate_risk(ticker='AAPL')"
  Option B — Final answer: "Based on the data, here is my analysis..."

The system:
  If LLM chose Option A → execute the tool, give result back to LLM
  If LLM chose Option B → return to user, done

This loop = the ReAct pattern from Day 1, now properly implemented.
C03

The ReAct Loop — Thought → Action → Observation

▼

The LLM orchestrator runs a loop that continues until the LLM decides it has enough information to give a final answer.

REACT LOOP — "COMPARE APPLE VS TESLA RISK AND SENTIMENT"
T1 Thought: I need risk data for both AAPL and TSLA. Let me call calculate_risk twice.
↓
A1 Action: LLM calls calculate_risk(ticker='AAPL') AND calculate_risk(ticker='TSLA')
↓
O1 Observation: AAPL → volatility 22.4%, MEDIUM. TSLA → volatility 58.7%, HIGH.
↓
T2 Thought: Now I need sentiment for both to complete the comparison.
↓
A2 Action: Calls get_stock_news('AAPL') AND get_stock_news('TSLA')
↓
O2 Observation: AAPL sentiment 0.68 (positive). TSLA sentiment 0.31 (neutral).
↓
✅ Final Answer: LLM synthesizes all 4 tool results into a coherent comparison paragraph.
THE KEY INSIGHT We never programmed "compare two stocks". The LLM figured out on its own that it needed 4 tool calls (2 risk + 2 news) across 2 iterations. That query had zero hardcoded logic. That's the power of LLM-driven orchestration.
C04

Memory Patterns — Episodic, Semantic, Procedural

▼
🌍 REAL WORLD — HOW A DOCTOR REMEMBERS THINGS A doctor uses three types of memory when treating patients — and your agent system maps exactly to the same three types.
📅

Episodic Memory

Specific past events, timestamped, tied to a session. "Mrs. Sharma came in last Tuesday with fever 102°F."
conversation:{session_id} in Redis
expires after 1hr inactivity
📚

Semantic Memory

General reusable knowledge that applies across sessions. "ACE inhibitors interact poorly with potassium."
cache:beta:TSLA in Redis
expires after 24 hours
⚙️

Procedural Memory

Stored routines — how to do things. "The procedure for taking blood pressure is..."
The LangGraph graph itself
lives in graph.py, not Redis

Build the LLM-Driven Orchestrator

WHY DESCRIPTIONS ARE EVERYTHING The description is literally how the LLM decides whether to call a tool — it reads it like a human reads a menu. Always state what to use it FOR and what NOT to use it for. Vague descriptions = wrong tool calls.
agents/tools.pypython
from langchain_core.tools import tool
# @tool decorator converts a regular Python function into a LangChain Tool
# that the LLM can call — name, description, and params are auto-extracted

@tool
def get_stock_price(ticker: str) -> str:
    """
    Fetches current stock price, daily change percentage, and RSI
    for the given ticker symbol from Yahoo Finance.

    Use this tool when the user asks about:
    - Current stock price or today's movement
    - RSI or momentum indicators
    - Whether a stock is overbought or oversold

    Do NOT use this for: news sentiment, volatility, or risk scoring.
    """
    from agents.price_agent import _fetch_price_sync
    from memory.cache import get_cached, set_cached

    cached = get_cached("price", ticker)
    if cached: return str(cached)
    result = _fetch_price_sync(ticker)
    set_cached("price", ticker, result)
    return str(result)


@tool
def get_stock_news(ticker: str) -> str:
    """
    Fetches recent news headlines and sentiment score for a stock.

    Use this tool when the user asks about:
    - Recent news about a company
    - Market sentiment or analyst opinions
    - Whether the news is positive or negative for a stock

    Do NOT use this for: price data, technical indicators, or risk metrics.
    """
    from tavily import TavilyClient
    from core.config import config

    tavily = TavilyClient(api_key=config.TAVILY_API_KEY)
    results = tavily.search(query=f"{ticker} stock news today", max_results=5)
    articles = results.get("results", [])
    headlines = [a.get("title","") for a in articles if a.get("title")]
    return str({"headlines": headlines[:3], "sentiment_score": 0.0})


@tool
def calculate_risk(ticker: str) -> str:
    """
    Calculates volatility, maximum drawdown, beta, and overall risk level
    for a given stock over the past 6 months.

    Use this tool when the user asks about:
    - How risky a stock is
    - Whether a stock is suitable for conservative/aggressive investors
    - Beta or market sensitivity

    Do NOT use this for: current price, news, or recommendations.
    """
    from agents.risk_agent import _calculate_risk_sync
    from memory.cache import get_cached, set_cached

    cached = get_cached("risk", ticker)
    if cached: return str(cached)
    result = _calculate_risk_sync(ticker)
    set_cached("risk", ticker, result)
    return str(result)


@tool
def get_full_analysis(ticker: str) -> str:
    """
    Runs a COMPLETE multi-agent analysis including price, news sentiment,
    risk metrics, and a BUY/HOLD/SELL recommendation.

    Use this tool when the user asks for:
    - A complete or full analysis
    - A recommendation on whether to buy, hold, or sell
    - A comprehensive overview of a stock

    This is the most expensive tool — only use when explicitly requested.
    """
    import json
    from agents.graph import build_graph
    graph = build_graph()
    state = {
        "ticker": ticker, "query_type": "full",
        # ... rest of None fields ...
        "errors": [], "retry_count": 0
    }
    final_state = graph.invoke(state)
    report = json.loads(final_state.get("final_report", "{}"))
    return str(report.get("verdict", {}))
agents/llm_orchestrator.pypython
from langchain_groq import ChatGroq
from langchain_core.messages import HumanMessage, SystemMessage, AIMessage, ToolMessage
# Message types represent the conversation history the LLM sees
# HumanMessage = what user said
# AIMessage    = what LLM responded (may include tool call request)
# ToolMessage  = result of a tool execution

from core.config import config
from agents.tools import get_stock_price, get_stock_news, calculate_risk, get_full_analysis

llm = ChatGroq(api_key=config.GROQ_API_KEY, model_name=config.MODEL_NAME)

# .bind_tools() tells the LLM "these are the tools available to you"
# LLM sees each tool's name + description and can pick autonomously
llm_with_tools = llm.bind_tools([
    get_stock_price, get_stock_news, calculate_risk, get_full_analysis
])


def run_llm_orchestrator(user_query: str, session_id: str = "default") -> str:
    """
    LLM-driven orchestrator — the LLM decides which tools to call.
    No hardcoded if/else routing. The LLM reads the query and reasons.
    """
    print(f"\n🤖 LLM Orchestrator: '{user_query}'")

    system_prompt = """You are an intelligent stock market analysis assistant.
You have tools for price data, news sentiment, risk metrics, and full analysis.

Rules:
1. Understand what the user is asking
2. Choose the MINIMUM set of tools needed for their specific question
3. Extract ticker symbols from company names (Apple=AAPL, Tesla=TSLA, Google=GOOGL)
4. Synthesize tool results into a clear, helpful answer"""

    messages = [
        SystemMessage(content=system_prompt),
        HumanMessage(content=user_query)
    ]

    # ── ReAct Loop ────────────────────────────────────────
    max_iterations = 5  # safety limit — prevents infinite loops

    for iteration in range(max_iterations):
        print(f"\n🔄 Iteration {iteration + 1}")

        response = llm_with_tools.invoke(messages)
        messages.append(response)
        # Add LLM's response to history so next iteration has full context

        if not response.tool_calls:
            # No tool calls = LLM decided it has enough info
            print(f"✅ Final answer ready")
            return response.content

        # LLM wants to call one or more tools
        for tool_call in response.tool_calls:
            tool_name     = tool_call["name"]
            tool_args     = tool_call["args"]
            tool_call_id  = tool_call["id"]
            # tool_call_id links this request to its result in the conversation

            print(f"🔧 LLM calling: {tool_name}({tool_args})")

            tool_map = {
                "get_stock_price":  get_stock_price,
                "get_stock_news":   get_stock_news,
                "calculate_risk":   calculate_risk,
                "get_full_analysis": get_full_analysis
            }

            tool_fn = tool_map.get(tool_name)
            try:
                tool_result = tool_fn.invoke(tool_args) if tool_fn else f"Unknown: {tool_name}"
                print(f"📊 Result: {str(tool_result)[:100]}...")
            except Exception as e:
                tool_result = f"Tool failed: {str(e)}"

            # Return result to LLM — ToolMessage links to the tool_call_id
            messages.append(ToolMessage(
                content=str(tool_result),
                tool_call_id=tool_call_id
            ))

    return "Analysis incomplete — maximum iterations reached."
api/main.py — add /chat endpointpython
from agents.llm_orchestrator import run_llm_orchestrator

class ChatRequest(BaseModel):
    query: str = Field(
        ...,
        description="Natural language query about any stock",
        examples=["Is Apple stock too risky for retirement portfolios?"]
    )
    session_id: str = Field(default="anonymous")

class ChatResponse(BaseModel):
    query:      str
    response:   str
    session_id: str

@app.post("/chat", response_model=ChatResponse, tags=["Chat"])
def chat(request: ChatRequest):
    """
    Natural language interface — ask anything about any stock.
    The LLM decides which tools to call based on your question.

    Examples:
    - "What is Tesla's current price?"
    - "Is Apple too volatile for conservative investors?"
    - "Give me a full analysis of Microsoft"
    - "Compare the risk and sentiment of Apple vs Tesla"
    """
    response = run_llm_orchestrator(request.query, request.session_id)
    return ChatResponse(query=request.query, response=response, session_id=request.session_id)

Now your API has two distinct endpoints for different use cases:

POST /analyze

  • → Structured — always runs full pipeline
  • → Returns defined JSON schema
  • → For programmatic / frontend consumption
  • → Fast, predictable, same output shape

POST /chat

  • → Conversational — any natural language query
  • → LLM picks tools + returns natural language
  • → For human users having a conversation
  • → Flexible, generalizes to any question

INTERVIEW

Q&A — Day 10

Q1What's the fundamental difference between your rule-based orchestrator (Day 5) and the LLM-driven one (Day 10)?▼
The rule-based orchestrator uses hardcoded Python if/else logic — if query_type is "quick" go here, if "full" go there. It only handles cases I explicitly programmed. The LLM-driven orchestrator gives the LLM a menu of tools with descriptions and lets it autonomously reason about which ones to call for any query. It generalizes to queries I never anticipated — like comparing two stocks or answering a nuanced risk question — without any new code.
Q2What makes a good tool description and why does it matter so much?▼
The description is literally how the LLM decides whether to call a tool — it reads the description the same way a human reads a menu. A vague description like "gets stock data" causes the LLM to call the wrong tool. A precise description that says exactly when TO use it and when NOT to use it ("use for price/RSI questions, NOT for news or risk") lets the LLM make correct decisions reliably. The description is the tool's "interface contract" with the LLM.
Q3Explain the ReAct loop that happens inside your LLM orchestrator.▼
The orchestrator runs a loop: first it sends the user's query to the LLM with the list of available tools. The LLM either returns a tool call request or a final text answer. If it's a tool call, we execute the tool, add the result back to the conversation history as a ToolMessage, and send everything back to the LLM for the next iteration. The LLM continues calling tools until it decides it has enough information, then produces a final answer. This is Thought → Action → Observation repeated until the LLM decides it's done.
Q4What are the three memory pattern types and how do they map to your system?▼
Episodic memory stores specific past events tied to a session — in my system that's conversation history stored in Redis with a session_id key, expiring after inactivity. Semantic memory stores general reusable facts — my cache layer storing computed beta values or risk scores for 24 hours since they don't change rapidly. Procedural memory encodes how to do things — my LangGraph graph structure itself is procedural memory, defining the procedure for conducting a stock analysis.
Q5What's the risk of allowing the LLM to call tools autonomously and how would you mitigate it?▼
The main risks are: the LLM calling expensive tools unnecessarily — mitigated by the max_iterations limit and precise tool descriptions that guide appropriate selection. The LLM could also extract the wrong ticker from an ambiguous query — mitigated by validation in each tool function. In production I'd add tool call logging for auditing, cost tracking per session to prevent abuse, and a human-in-the-loop confirmation for high-stakes decisions like trade execution.
Q6How does your /chat endpoint differ from /analyze and when would each be used?▼
The /analyze endpoint is structured — it always runs the full pipeline for a specific ticker and returns a defined JSON schema. It's designed for programmatic consumption by a frontend or downstream system that needs specific fields. The /chat endpoint is conversational — it accepts any natural language query, lets the LLM decide what data to fetch, and returns a natural language answer. /analyze is for systems consuming structured data; /chat is for human users having a conversation.
