Core Ideas — Day 13

C01

Why Testing Agent Systems is Uniquely Challenging

▼
🌍 REAL WORLD — REGULAR CAR vs SELF-DRIVING CAR Regular car: press brake → car stops. Predictable, deterministic, easy to test.

Self-driving car: show a stop sign → it should stop. But what if it's covered in snow? What if a pedestrian is jaywalking? The car makes DECISIONS — testing decisions is harder than testing mechanical responses.

Your agent system is the self-driving car.
CHALLENGE 1
Non-Determinism
LLM might say "BUY" one run and "HOLD" the next for same data. → Assert output is in ["BUY","HOLD","SELL"], not assert exact string.
CHALLENGE 2
External Dependencies
yFinance, Tavily, Groq require internet, are slow, and cost money. → MOCK these dependencies to make tests fast, free, and reliable.
CHALLENGE 3
Complex State Flows
A bug in Risk Agent only shows up when Analyst Agent reads bad State. → Integration tests that run multiple agents together.
CHALLENGE 4
Infrastructure Dependencies
Redis must be running for cache tests. → Separate test categories: some mock Redis, some need real Redis.
C02

Unit vs Integration vs End-to-End Tests

▼
🌍 REAL WORLD — TESTING A BICYCLE Unit Test: Test brakes alone — squeeze lever → pads engage. Fast, isolated, catches exactly where the problem is.
Integration Test: Test gear + chain + wheel together — shift gear → wheel speed changes. Catches "works alone, breaks together" bugs.
End-to-End Test: Person rides 5km with hills and turns. Closest to real user experience, but slowest and most expensive.
test_categories.txttext
UNIT TESTS (fast, isolated — ~0.1s each):
  → Test _fetch_price_sync() with MOCKED yFinance
  → Test RSI calculation formula with known input/output
  → Test validate_and_resolve_ticker() with various strings
  → No internet. No Redis. No API keys.

INTEGRATION TESTS (medium — ~1-2s each):
  → Test Price Agent + Cache working together (real Redis)
  → Test Orchestrator routing → correct agent is called
  → Test State correctly flows from Price Agent to Analyst Agent

END-TO-END TESTS (slow — 10s+ each):
  → Full /analyze endpoint with real AAPL from Yahoo Finance
  → Full pipeline: Orchestrator → Price → News → Risk → Analyst → Report
  → Run sparingly — once before deployment only
  → These cost real API money
C03

Mocking — Flight Simulators for Code

▼
🌍 REAL WORLD — PILOT FLIGHT SIMULATORS Real plane: $5000/hour, dangerous, can't test engine failure safely.
Simulator: $50/hour, safe, available anytime, CAN specifically test emergencies.

The simulator gives the SAME INTERFACE as a real plane but with CONTROLLED, PREDICTABLE responses. That's exactly what mocking does for your tests.

C04

pytest — Key Features You'll Use

▼

@pytest.fixture

Reusable setup code injected into tests by name. Define once in conftest.py, use everywhere.

def test_agent(base_state): ...

@pytest.mark.parametrize

Run the same test with multiple inputs automatically. One test function covers many cases.

@parametrize("ticker", ["AAPL","TSLA"])

pytest.approx()

Floating point comparison with tolerance. 0.1+0.2 ≠ 0.3 exactly — approx handles this.

assert result == pytest.approx(10.0, abs=0.01)

@pytest.mark.asyncio

Lets pytest run async test functions. Required for testing your async agent nodes.

async def test_agent_node(): ...

# Test a function raises the right error
with pytest.raises(ValueError) as exc_info:
    validate_and_resolve_ticker("")

@pytest.mark.parametrize("ticker,valid", [
    ("AAPL", True), ("TSLA", True),
    ("",     False), ("INVALIDXYZ", False),
])
def test_ticker_validation(ticker, valid):
    resolved, error = validate_and_resolve_ticker(ticker)
    assert (error is None) == valid

import pytest
import pandas as pd
from unittest.mock import patch
from agents.price_agent import _fetch_price_sync, price_agent_node


class TestFetchPriceSync:
    """Unit tests for _fetch_price_sync — mocked yFinance."""

    def test_returns_correct_keys(self, mock_yfinance_data):
        with patch("agents.price_agent.yf.Ticker") as mock_ticker:
            # patch() temporarily replaces yf.Ticker with a mock object
            mock_ticker.return_value.history.return_value = mock_yfinance_data
            result = _fetch_price_sync("AAPL")
            assert "current_price" in result
            assert "price_change_pct" in result
            assert "rsi" in result

    def test_rsi_within_valid_range(self, mock_yfinance_data):
        """RSI must always be between 0 and 100 — that's its definition."""
        with patch("agents.price_agent.yf.Ticker") as mock_ticker:
            mock_ticker.return_value.history.return_value = mock_yfinance_data
            result = _fetch_price_sync("AAPL")
            assert 0 <= result["rsi"] <= 100

    def test_empty_dataframe_raises_error(self):
        """If yFinance returns empty data, should raise ValueError."""
        with patch("agents.price_agent.yf.Ticker") as mock_ticker:
            mock_ticker.return_value.history.return_value = pd.DataFrame()
            with pytest.raises(ValueError) as exc_info:
                _fetch_price_sync("INVALIDXYZ")
            assert "No price data" in str(exc_info.value)

    def test_price_change_calculation(self):
        """Verify 10% increase is calculated correctly."""
        df = pd.DataFrame({"Close": [100.0, 110.0]})
        with patch("agents.price_agent.yf.Ticker") as mock_ticker:
            mock_ticker.return_value.history.return_value = df
            result = _fetch_price_sync("TEST")
            assert result["price_change_pct"] == pytest.approx(10.0, abs=0.01)
            # pytest.approx() handles floating point precision issues


class TestPriceAgentNode:
    """Async integration tests for the full price_agent_node."""

    @pytest.mark.asyncio
    async def test_cache_hit_skips_yfinance(self, base_state):
        """If cache has data, yFinance should NEVER be called."""
        cached = {"current_price": 189.3, "price_change_pct": 1.2, "rsi": 58.4}
        with patch("agents.price_agent.get_cached", return_value=cached):
            with patch("agents.price_agent.yf.Ticker") as mock_yf:
                result = await price_agent_node(base_state)
                mock_yf.assert_not_called()  # ← verify yFinance was never touched
                assert result["current_price"] == 189.3

    @pytest.mark.asyncio
    async def test_yfinance_failure_returns_errors(self, base_state):
        """If yFinance crashes, node returns errors gracefully — doesn't raise."""
        with patch("agents.price_agent.get_cached", return_value=None):
            with patch("agents.price_agent.yf.Ticker") as mock_yf:
                mock_yf.side_effect = Exception("Yahoo Finance is down")
                # side_effect = raise this exception when the mock is called
                result = await price_agent_node(base_state)
                assert result["current_price"] is None
                assert len(result["errors"]) > 0
                assert "Yahoo Finance is down" in result["errors"][0]
tests/test_risk_agent.pypython
import pytest
import pandas as pd
import numpy as np
from unittest.mock import patch


class TestRiskCalculations:

    def test_zero_volatility_stock(self):
        """Constant price = zero daily returns = zero volatility"""
        from agents.risk_agent import _calculate_risk_sync
        closes = pd.Series([100.0] * 100)
        df     = pd.DataFrame({"Close": closes})

        with patch("agents.risk_agent.yf.Ticker") as mock_ticker:
            mock_ticker.return_value.history.side_effect = [df, df]
            # side_effect=[a,b] → first call returns a, second returns b
            result = _calculate_risk_sync("AAPL")
            assert result["volatility"] == pytest.approx(0.0, abs=0.1)

    def test_high_volatility_classified_high_risk(self):
        """Stocks with >40% annual volatility → HIGH risk level"""
        from agents.risk_agent import _calculate_risk_sync

        # Alternating +10%/-10% every day = extreme volatility
        prices = []
        price  = 100.0
        for i in range(200):
            price *= 1.10 if i % 2 == 0 else 0.90
            prices.append(price)

        df     = pd.DataFrame({"Close": pd.Series(prices)})
        spy_df = pd.DataFrame({"Close": pd.Series([100.0] * 200)})

        with patch("agents.risk_agent.yf.Ticker") as mock_ticker:
            mock_ticker.return_value.history.side_effect = [df, spy_df]
            result = _calculate_risk_sync("TSLA")
            assert result["risk_level"] == "HIGH"

    def test_risk_level_always_valid(self):
        """risk_level must always be one of exactly three values"""
        from agents.risk_agent import _calculate_risk_sync
        closes = pd.Series(np.linspace(100, 110, 200))
        df     = pd.DataFrame({"Close": closes})

        with patch("agents.risk_agent.yf.Ticker") as mock_ticker:
            mock_ticker.return_value.history.side_effect = [df, df]
            result = _calculate_risk_sync("AAPL")
            assert result["risk_level"] in ["LOW", "MEDIUM", "HIGH"]

    def test_max_drawdown_is_never_positive(self):
        """Drawdown = loss from peak — mathematically must be ≤ 0"""
        closes       = pd.Series(np.linspace(100, 110, 200))
        daily_returns = closes.pct_change().dropna()
        cumulative    = (1 + daily_returns).cumprod()
        rolling_max   = cumulative.cummax()
        drawdown      = (cumulative - rolling_max) / rolling_max
        max_drawdown  = float(drawdown.min()) * 100
        assert max_drawdown <= 0

Q&A — Day 13

Q1Why is testing AI agent systems more challenging than testing regular software?▼
Three main reasons: first, LLMs are non-deterministic — the same prompt can return different outputs on different runs, so you can't assert exact LLM responses, only that outputs conform to expected shapes and value ranges. Second, agents depend on external APIs that are slow, costly, and can fail, requiring mocking to make tests fast and reliable. Third, bugs in multi-agent systems are often emergent — an individual agent looks correct in isolation but produces wrong output when State from a previous agent is malformed.
Q2What is mocking and why is it essential for testing your agent system?▼
Mocking replaces a real dependency with a fake version that returns controlled, predictable outputs. In my tests, I mock yFinance to return a pre-defined DataFrame instead of hitting Yahoo Finance's servers — making tests run in 0.001 seconds instead of 2-3 seconds, without requiring internet, without costing API credits, and allowing me to test edge cases like empty data or network failures that I can't easily trigger with real APIs.
Q3What's the difference between unit, integration, and end-to-end tests?▼
Unit tests verify a single function in complete isolation — like testing the RSI calculation formula with known inputs. Integration tests verify components working together — like testing Price Agent plus the cache layer to confirm data is actually stored and retrieved correctly. End-to-end tests exercise the full system under realistic conditions — making a real API call through all 5 agents. Each level catches different bugs; I prioritize unit and integration tests for speed, with end-to-end tests reserved for pre-deployment verification.
Q4What does pytest.approx() do and why do you need it for floating point assertions?▼
Floating point arithmetic has tiny precision errors — 0.1 + 0.2 gives 0.30000000000000004, not exactly 0.3. If I assert result == 10.0 for a percentage calculation, it might fail due to microscopic rounding differences even when the logic is correct. pytest.approx(10.0, abs=0.01) asserts that the result is within ±0.01 of 10.0, which correctly passes for floating point results while still catching real logic errors.
Q5How does conftest.py work and why put fixtures there instead of in each test file?▼
conftest.py is a special pytest file automatically discovered and loaded before any tests run. Fixtures defined there are available to all test files in the same directory without any imports — pytest injects them by matching the fixture name to the test function's parameter name. I put shared fixtures like base_state and mock_yfinance_data there because they're used across multiple test files. Duplicating them in each file would violate DRY and mean updating the same fixture in multiple places when the State schema changes.
Q6How would you test the LLM-driven orchestrator from Day 10 given non-deterministic LLM output?▼
I'd test around the LLM, not through it. Unit tests would mock the LLM's response to return a known tool call decision and verify the tool execution logic works correctly. For the actual LLM behavior, I'd use contract tests — run a small set of representative queries with the real LLM and assert outputs satisfy structural constraints: recommendation is in BUY/HOLD/SELL, confidence is between 0 and 1, the response is non-empty. This tests "LLM integration is working" without requiring deterministic outputs.
