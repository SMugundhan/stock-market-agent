🧠 Concept 1

📋
What is State?

The Shared Patient File
The backbone of every LangGraph system — how agents share data
🌍 Real World Scenario — Hospital

When a patient comes in, every department — doctor, nurse, lab, pharmacist — reads and writes to the same patient file. Nobody carries their own separate copy. The file is always up to date for whoever needs it next.

That patient file = State in LangGraph.

Price Agent → WRITES current_price, rsi, price_change_pct News Agent → WRITES news_headlines, sentiment_score Risk Agent → WRITES volatility, risk_level Analyst Agent → READS price + news + risk → WRITES recommendation Report Agent → READS everything → WRITES final_report
📈 Price Agent
Writes:
current_price rsi price_change_pct
📰 News Agent
Writes:
news_headlines sentiment_score
⚠️ Risk Agent
Writes:
volatility risk_level
🧠 Analyst Agent
Reads:
price news risk
Writes:
recommendation
💡
Think of State as a baton in a relay race. Each runner (agent) adds something to the baton and passes it on. By the finish line it contains everything from every runner.
🧠 Concept 2

⬡
What are Nodes?

Workstations in a Car Factory
Each node does one specific job — nothing more, nothing less
🌍 Real World Scenario — Car Factory

Station 1 fits the engine. Station 2 fits the doors. Station 3 paints the car. Each station does one specific job and passes the car to the next. Each station = a Node.

A Node is just a Python function that:

1. Receives the current State 2. Does its job (fetch data / call LLM / calculate something) 3. Returns a dict of updated State fields (LangGraph merges this automatically)
python — anatomy of a node
def price_agent_node(state: StockAnalysisState) -> dict:
    # ① Read from state
    ticker = state["ticker"]

    # ② Do the job
    price_data = fetch_stock_price(ticker)

    # ③ Return ONLY updated fields
    # LangGraph merges this dict with existing state automatically
    return {"current_price": price_data}
🎯
Each node is completely isolated — it only knows about State, not about other nodes. This makes them independently testable and replaceable.
🧠 Concept 3

🔀
What are Edges?

Roads Between Cities
Edges define how data flows from one node to the next
🌍 Real World Scenario — Roads

Mumbai → Pune — always use this route. (Normal Edge)
Delhi → Jaipur OR Agra — depends on traffic. (Conditional Edge)

Normal Edge Fixed
Always routes to the same next node. Used when the flow is predictable.
python
graph.add_edge(
  "price_agent",
  "analyst_agent"
)
# price ALWAYS → analyst
Conditional Edge Dynamic
A function decides the next node based on State. Enables loops and branching.
python
def should_retry(state):
  if state["confidence"] < 0.5:
    return "retry"
  return "report_agent"

graph.add_conditional_edges(
  "analyst_agent", should_retry
)
⚡
Conditional edges are what make LangGraph truly powerful — agents can loop, retry, branch, or escalate based on what they find. This is impossible in a simple chain.

♻️
The Graph Lifecycle

How LangGraph Executes Your System
From definition to final result — step by step
1
Define State Schema
TypedDict that describes all data flowing through the system
2
Create Nodes
Python functions — each does one specific job
3
Connect with Edges
Define how data flows — normal or conditional
4
Set Entry Point
Which node runs first when graph is invoked
5
Compile
Validates graph structure, checks all edges connect to real nodes
6
Invoke with Initial State
Pass starting data — LangGraph runs node by node, updating State each step
✓
Reaches END → Returns Final State
All agents have run, all fields are filled, result is ready
🎲
Board game analogy: Board = Graph | Pieces = State | Squares = Nodes | Rules = Edges | Win condition = END node

TypedDict — State Schema

The Blueprint of Your Data
Documents every field, its type, and which agent fills it
python — core/state.py
from typing import TypedDict, List, Optional

class StockAnalysisState(TypedDict):
    # ── INPUT ─────────────────────────────────────
    ticker: str                   # "AAPL" — user provides this

    # ── PRICE AGENT fills these ───────────────────
    current_price:    Optional[float]      # e.g. 189.3
    price_change_pct: Optional[float]      # e.g. +1.2%
    rsi:              Optional[float]      # 0-100, above 70 = overbought

    # ── NEWS AGENT fills these ────────────────────
    news_headlines:   Optional[List[str]]  # ["Apple hits record...", ...]
    sentiment_score:  Optional[float]      # -1.0 to +1.0

    # ── RISK AGENT fills these ────────────────────
    volatility:       Optional[float]      # higher = riskier
    risk_level:       Optional[str]        # "LOW" / "MEDIUM" / "HIGH"

    # ── ANALYST AGENT fills these ─────────────────
    recommendation:   Optional[str]        # "BUY" / "HOLD" / "SELL"
    confidence:       Optional[float]      # 0.0 to 1.0
    reasoning:        Optional[str]        # 2-3 sentence explanation

    # ── REPORT AGENT fills this ───────────────────
    final_report:     Optional[str]        # full natural language report

    # ── METADATA ──────────────────────────────────
    errors:           Optional[List[str]]  # any errors during processing
Why TypedDict?What it gives you
Documents data flowAnyone reading code immediately knows what flows through the system
Type safetyCatches bugs early if agent returns wrong data type
Optional fieldsAgents only fill their own fields — others stay None until filled
Interview signalShows you think about data contracts — senior engineer behavior

agents/price_agent.py — Fetch Real Stock Data + RSI
Uses yFinance to get live AAPL data. Calculates RSI manually so you truly understand it.
python — agents/price_agent.py
import yfinance as yf
# yfinance = Yahoo Finance's free Python library for real stock data

from core.state import StockAnalysisState

def price_agent_node(state: StockAnalysisState) -> dict:

    ticker = state["ticker"]
    print(f"📈 Price Agent: Fetching {ticker}...")

    try:
        stock = yf.Ticker(ticker)
        # yf.Ticker("AAPL") = object connected to AAPL's data on Yahoo Finance

        hist = stock.history(period="1mo", interval="1d")
        # period="1mo" = last 1 month of data
        # interval="1d" = one row per day (daily candles)
        # Returns DataFrame: Open, High, Low, Close, Volume columns

        current_price = round(float(hist["Close"].iloc[-1]), 2)
        # iloc[-1] = last row = most recent day's closing price

        prev_price = float(hist["Close"].iloc[-2])
        # iloc[-2] = second last row = yesterday's close

        price_change_pct = round(((current_price - prev_price) / prev_price) * 100, 2)
        # Formula: ((today - yesterday) / yesterday) × 100

        # ── RSI Calculation ───────────────────────────────────
        closes = hist["Close"]
        delta = closes.diff()           # daily price changes
        gain  = delta.clip(lower=0)    # keep only positive changes
        loss  = -delta.clip(upper=0)   # keep only negative (make positive)

        avg_gain = gain.rolling(window=14).mean().iloc[-1]
        avg_loss = loss.rolling(window=14).mean().iloc[-1]
        # 14-day rolling average = industry standard RSI window

        if avg_loss == 0:
            rsi = 100.0             # no losses at all = maximum overbought
        else:
            rs  = avg_gain / avg_loss  # RS = Relative Strength ratio
            rsi = round(100 - (100 / (1 + rs)), 2)
            # RSI formula: 100 - (100 / (1 + RS))
            # Above 70 = overbought | Below 30 = oversold | 30-70 = neutral

        print(f"✅ ${current_price} | {price_change_pct}% | RSI: {rsi}")

        return {
            "current_price":    current_price,
            "price_change_pct": price_change_pct,
            "rsi":              rsi,
            "errors": []
        }

    except Exception as e:
        print(f"❌ Price Agent Error: {e}")
        return {
            "current_price": None, "price_change_pct": None,
            "rsi": None, "errors": [f"Price Agent failed: {str(e)}"]
        }

agents/analyst_agent.py — LLM Recommendation
Feeds real price data into LLaMA 3.1. This is why agents beat simple prompts — we give actual data, not guesses.
python — agents/analyst_agent.py
from langchain_groq import ChatGroq
from core.state  import StockAnalysisState
from core.config import config

llm = ChatGroq(api_key=config.GROQ_API_KEY, model_name=config.MODEL_NAME)
# Initialize once at module level — more efficient than creating inside function

def analyst_agent_node(state: StockAnalysisState) -> dict:

    print(f"🧠 Analyst Agent: Analyzing {state['ticker']}...")

    prompt = f"""You are a senior stock market analyst.
Analyze this real-time data and give a recommendation:

Stock:        {state['ticker']}
Price:        ${state.get('current_price', 'N/A')}
Change:       {state.get('price_change_pct', 'N/A')}%
RSI:          {state.get('rsi', 'N/A')}

RSI Guide: Above 70 = overbought | Below 30 = oversold | 30-70 = neutral

Respond in EXACTLY this format:
RECOMMENDATION: [BUY/HOLD/SELL]
CONFIDENCE: [0.0 to 1.0]
REASONING: [2-3 sentences]"""
    # Structured prompt = structured output = easy to parse
    # We feed REAL data — agent reasons over facts, not training data

    try:
        response = llm.invoke(prompt)
        raw_text = response.content
        # response.content = the text LLaMA returned

        lines = raw_text.strip().split('\n')
        # split('\n') breaks text into list of lines for easy parsing

        recommendation, confidence, reasoning = "HOLD", 0.5, raw_text
        # defaults in case parsing fails

        for line in lines:
            if line.startswith("RECOMMENDATION:"):
                recommendation = line.split(":")[1].strip()
            elif line.startswith("CONFIDENCE:"):
                confidence = float(line.split(":")[1].strip())
            elif line.startswith("REASONING:"):
                reasoning = line.split(":", 1)[1].strip()
                # split(":", 1) = split on FIRST ":" only
                # reasoning text might contain ":" so we only split once

        print(f"✅ {recommendation} (confidence: {confidence})")
        return {"recommendation": recommendation, "confidence": confidence, "reasoning": reasoning}

    except Exception as e:
        return {"recommendation": "HOLD", "confidence": 0.0, "reasoning": f"Failed: {e}"}

agents/graph.py — Wire the Graph
Connect the two nodes with edges. Set entry point. Compile. This is the LangGraph "wiring."
python — agents/graph.py
from langgraph.graph import StateGraph, END
# StateGraph = main class to build your graph
# END       = special constant that marks where the graph stops

from core.state             import StockAnalysisState
from agents.price_agent    import price_agent_node
from agents.analyst_agent  import analyst_agent_node

def build_graph():

    graph = StateGraph(StockAnalysisState)
    # Tells LangGraph: "data flowing through this graph follows StockAnalysisState"

    graph.add_node("price_agent",   price_agent_node)
    graph.add_node("analyst_agent", analyst_agent_node)
    # add_node(name, function) — name is used in edges

    graph.set_entry_point("price_agent")
    # Which node runs FIRST when graph is invoked

    graph.add_edge("price_agent", "analyst_agent")
    # Normal edge — price_agent ALWAYS goes to analyst_agent next

    graph.add_edge("analyst_agent", END)
    # After analyst finishes, graph is done

    return graph.compile()
    # compile() validates structure + prepares execution engine
    # Like "building" a program before running it

from agents.graph import build_graph

def main():
    graph = build_graph()
    print("✅ Graph compiled\n")

    initial_state = {
        "ticker": "AAPL",      # only this is filled at start
        "current_price":    None, # Price Agent will fill
        "price_change_pct": None, # Price Agent will fill
        "rsi":              None, # Price Agent will fill
        "news_headlines":   None, # News Agent — Day 3
        "sentiment_score":  None, # News Agent — Day 3
        "volatility":       None, # Risk Agent — Day 4
        "risk_level":       None, # Risk Agent — Day 4
        "recommendation":   None, # Analyst Agent will fill
        "confidence":       None, # Analyst Agent will fill
        "reasoning":        None, # Analyst Agent will fill
        "final_report":     None, # Report Agent — Day 7
        "errors": []
    }

    final_state = graph.invoke(initial_state)
    # .invoke() runs the full graph synchronously
    # Returns FINAL state after all nodes have executed

    print("\n📊 FINAL RESULTS")
    print(f"Price:          ${final_state['current_price']}")
    print(f"Change:         {final_state['price_change_pct']}%")
    print(f"RSI:            {final_state['rsi']}")
    print(f"Recommendation: {final_state['recommendation']}")
    print(f"Confidence:     {final_state['confidence']}")
    print(f"Reasoning:      {final_state['reasoning']}")

if __name__ == "__main__":
    # Only runs main() if this file is run directly
    # If someone imports this file, main() won't auto-run
    main()

Interview Questions & Answers

Q1 What is State in LangGraph and why is it important?
State is a shared data object passed through every node in the graph. Each agent reads from it and writes its outputs back to it — like a hospital patient file that every department updates. Without State, agents would have no way to share data or build on each other's work.
Q2 What is a Node in LangGraph? How simple or complex can it be?
A Node is simply a Python function that receives current State, does its job, and returns a dict of updated fields. LangGraph merges the returned dict automatically. It can be as simple as a calculation or as complex as an LLM call with tool use. Each node is isolated — it only knows about State, not about other nodes.
Q3 What are the two types of Edges? When would you use each?
Normal edges always route to the same next node — used when flow is fixed and predictable (e.g., price_agent always feeds analyst_agent). Conditional edges use a function to decide based on State — used for loops, retries, or branching (e.g., if confidence is low, retry analysis; otherwise proceed to report).
Q4 Why use TypedDict for State definition?
TypedDict documents the full data contract of the system — every field, its type, and which agent fills it. It catches bugs early if an agent returns the wrong type, makes code self-documenting, and in a team setting anyone can look at the State and immediately understand the entire data flow. It shows production-level thinking.
Q5 What does graph.compile() do and why is it needed?
compile() validates the graph structure — checks that all edges connect to real nodes, entry points are set, and the graph can reach END. It also prepares the execution engine. Without compiling, you cannot invoke the graph. Think of it like compiling a program before running it — it catches structural errors upfront.
Q6 How do you handle errors in individual agents without crashing the whole pipeline?
Each node has a try/except block. On failure it returns None for its output fields and appends the error message to the errors list in State. This way one agent failing doesn't crash the entire graph — downstream agents can check the errors field and handle missing data gracefully. The orchestrator can also inspect errors and decide whether to retry or return a partial result.
